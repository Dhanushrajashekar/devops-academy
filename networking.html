OSI Model — The 7 Layers

The framework that describes how data moves from one computer to another. Every networking interview starts here.

Critical~25 min
🧠
Memory trick: "Please Do Not Throw Sausage Pizza Away"
Physical, Data Link, Network, Transport, Session, Presentation, Application — layers 1 to 7.
7
Application
The layer you interact with. Provides network services to applications.
HTTP, HTTPS, FTP, SMTP, DNS, SSH, DHCP, WebSocket
6
Presentation
Data formatting, encryption, compression. Translates between app and network formats.
TLS/SSL encryption, JPEG/PNG/MPEG encoding, JSON/XML serialisation
5
Session
Establishes, manages, and terminates sessions between applications.
NetBIOS, RPC, SQL sessions, authentication sessions
4
Transport
End-to-end communication. Reliability, flow control, segmentation.
TCP (reliable, ordered), UDP (fast, no guarantee) — ports live here
3
Network
Logical addressing and routing. Gets packets from source to destination across networks.
IP (IPv4, IPv6), ICMP (ping), routers, routing tables
2
Data Link
Node-to-node delivery on the same network. MAC addresses, error detection.
Ethernet, Wi-Fi (802.11), MAC addresses, switches, ARP
1
Physical
Raw bits over a physical medium. Cables, signals, hardware.
Ethernet cables (Cat5e/6), fibre optic, Wi-Fi radio waves, hubs, repeaters

What Layer Does Each Device Operate At?

DeviceOSI LayerWhat It Does
HubLayer 1 — PhysicalBroadcasts all traffic to all ports. Dumb. No intelligence.
SwitchLayer 2 — Data LinkForwards frames using MAC address table. Sends traffic only to the right port.
RouterLayer 3 — NetworkRoutes packets between different networks using IP addresses and routing tables.
FirewallL3–L7Filters traffic. Basic = Layer 3/4 (IP + port). Next-gen = Layer 7 (application content).
Load BalancerLayer 4 or 7L4: distributes TCP connections. L7 (ALB): routes based on HTTP path/headers.
🎯
Interview: "At what layer does a router operate, and why?"
Answer: Layer 3 — Network. A router uses IP addresses (Layer 3) to decide where to forward packets. It reads the destination IP, consults its routing table, and forwards out the correct interface. It doesn't care about MAC addresses (Layer 2) or what application generated the traffic (Layer 7).

🎯
Interview: "Why does DNS use UDP instead of TCP?"
Answer: DNS queries are tiny (usually under 512 bytes) and need to be fast. The overhead of a TCP 3-way handshake would add latency. UDP lets the client send the query and get an answer in one round-trip. If the response is too large (e.g., DNSSEC records, zone transfers), DNS falls back to TCP automatically.

🎯
Interview: "What's the difference between HTTP and HTTPS?"
HTTP sends data in plaintext — anyone on the network (ISP, coffee shop Wi-Fi, MITM attacker) can read it. HTTPS wraps HTTP inside TLS, which provides: Encryption (data unreadable in transit), Authentication (you're actually talking to example.com, not an impersonator), Integrity (data can't be tampered with in transit). TLS runs on port 443; HTTP on port 80.

🌍 What Happens When You Type a URL?

The single most common networking interview question. Know every step — DNS, TCP, TLS, HTTP — end to end.

Must Know~30 min
🎯
Interview tip
This question is asked at Google, Amazon, Cloudflare, and almost every DevOps role. Go layer by layer. The interviewer wants to see how deep you can go.

Step-by-Step: https://www.example.com/page

Step 1 — Browser Cache & OS Cache Check

Before any network request, the browser checks its own DNS cache (stored from recent visits). If found, jumps straight to TCP. Then it checks the OS-level DNS cache (/etc/hosts on Linux, the Windows DNS cache service). If either has a fresh record (within TTL), DNS resolution is skipped entirely.

Step 2 — DNS Resolution

If not cached, the OS asks the configured DNS resolver (usually your router or ISP, or a corporate DNS server). That resolver performs a recursive lookup:

Client → Recursive Resolver (8.8.8.8) → Root Nameserver ("who handles .com?") → .com TLD Server ("ns1.example.com handles example.com") → Authoritative Nameserver (ns1.example.com) ← Returns: www.example.com → 93.184.216.34 (TTL: 3600s) Client caches this for TTL duration

The resolver caches the answer. Your OS caches it. Your browser caches it. Future requests within TTL skip all these steps.

Step 3 — ARP (if on same network)

To send packets to the default gateway (your router), your machine needs the router's MAC address. It checks its ARP cache. If missing, it broadcasts: "Who has 192.168.1.1?" The router replies with its MAC. Now packets can be sent at Layer 2.

Step 4 — TCP 3-Way Handshake

TCP is connection-oriented — a connection must be established before any data is sent. Three packets are exchanged:

Client → Server: SYN (seq=100) "I want to connect" Server → Client: SYN-ACK (seq=200, ack=101) "OK, I'm ready" Client → Server: ACK (ack=201) "Great, connected!" ─── Connection established in ~1 RTT ───

RTT (Round-Trip Time) is the delay for one packet to travel to the server and back. A connection to a London server from India might cost 100ms just for the handshake.

Step 5 — TLS Handshake (HTTPS only)

Because the URL is https://, TLS runs on top of TCP before any HTTP data is sent:

Client → Server: ClientHello (TLS version, cipher list, random bytes) Server → Client: ServerHello + Certificate + ServerHelloDone Client: Validates certificate (CA trust chain, expiry, hostname match) Client → Server: Pre-master secret (encrypted with server's public key) Both sides: Derive same symmetric session key independently Client → Server: "Finished" (encrypted) Server → Client: "Finished" (encrypted) ─── All future data is encrypted with symmetric key ───

Modern TLS 1.3 does this in 1 RTT (instead of TLS 1.2's 2 RTTs) by combining steps. The session key is never transmitted — both sides derive it from shared random values (Diffie-Hellman key exchange).

Step 6 — HTTP Request

Now the browser sends an HTTP GET request through the encrypted tunnel:

GET /page HTTP/1.1 Host: www.example.com Accept: text/html,application/xhtml+xml Accept-Encoding: gzip, deflate, br Accept-Language: en-US,en;q=0.9 Connection: keep-alive Cookie: session_id=abc123

Step 7 — Server-Side Processing

The request may travel through several hops before reaching the application:

Internet → CDN Edge Node (CloudFront/Cloudflare) → Load Balancer (AWS ALB / nginx) → Web Server (nginx / Apache) → Application Server (Node.js / Python / Java) → Database (PostgreSQL / Redis cache) → Response assembled

If the CDN has a cached copy of /page, the request never reaches your server at all — the CDN serves it from the nearest edge location worldwide.

Step 8 — HTTP Response

HTTP/1.1 200 OK Content-Type: text/html; charset=UTF-8 Content-Encoding: gzip Content-Length: 4823 Cache-Control: max-age=3600 Set-Cookie: session_id=abc123; HttpOnly; Secure <!DOCTYPE html>... (gzip-compressed HTML body)

Step 9 — Browser Renders the Page

The browser parses HTML → builds DOM, parses CSS → builds CSSOM, combines into Render Tree, runs JavaScript. Sub-resources (images, CSS files, JS files) each trigger their own DNS+TCP+TLS+HTTP cycles — but HTTP/2 multiplexes multiple requests over a single TCP connection so they happen in parallel.

ARP — Address Resolution Protocol

Bridges the gap between Layer 3 (IP addresses) and Layer 2 (MAC addresses). Without ARP, packets can't leave your network.

Important~20 min

Why ARP Exists

IP addresses are logical (they can change). MAC addresses are physical (burned into hardware). When you send a packet to 192.168.1.50, the OS needs to know which physical NIC on the local network segment has that IP — that's what ARP resolves.

ARP Request/Reply Process

# Your machine wants to send to 192.168.1.50 # First checks ARP cache — not found # Broadcasts to everyone on the subnet: ARP Request (broadcast): "Who has 192.168.1.50? Tell 192.168.1.10" Destination MAC: FF:FF:FF:FF:FF:FF (broadcast — everyone receives) Source MAC: AA:BB:CC:DD:EE:FF (your MAC) ARP Reply (unicast — only the target responds): "192.168.1.50 is at 11:22:33:44:55:66" Destination MAC: AA:BB:CC:DD:EE:FF (your MAC) # Now your machine caches this mapping # View ARP cache: arp -a ip neigh show

ARP Cache

ARP entries are cached to avoid broadcasting on every packet. They expire after a timeout (typically 20-120 seconds on Linux). You can view and manipulate the cache:

ip neigh show # View ARP cache (modern) arp -a # View ARP cache (legacy) ip neigh flush all # Clear ARP cache ip neigh add 192.168.1.50 lladdr 11:22:33:44:55:66 dev eth0 # Static entry

Gratuitous ARP

A device sends an ARP reply without being asked. Used to: (1) announce a new IP, (2) update neighbours after failover, (3) detect IP conflicts. Load balancers use gratuitous ARP during failover so all devices on the network immediately route to the new primary.

Proxy ARP

A router responds to ARP requests on behalf of hosts in another network. Allows hosts without a configured default gateway to reach remote hosts. Rarely used today but appears in legacy networks.

ARP Spoofing (Security)

An attacker sends fake ARP replies to poison the ARP cache of victims:

# Attacker tells Victim: "192.168.1.1 (the router) is at MY MAC" # Attacker tells Router: "192.168.1.100 (the victim) is at MY MAC" # Result: all traffic between Victim ↔ Router flows through attacker (MITM) # Mitigations: # - Dynamic ARP Inspection (DAI) on managed switches # - Static ARP entries for critical hosts # - 802.1X port authentication # - Use HTTPS/TLS — even if MITM'd, traffic is encrypted

Routing — How Packets Travel Across Networks

Routers are the post offices of the internet. Understanding routing tables, static vs dynamic routes, and BGP is essential for cloud and network roles.

Critical~30 min

What a Router Does

A router operates at Layer 3 (Network). For each incoming packet it reads the destination IP, looks up the best matching route in its routing table, and forwards the packet out the appropriate interface — decrementing the IP TTL field by 1. If TTL hits 0, the packet is dropped (prevents infinite loops).

Routing Table

# View the routing table on Linux ip route show # Typical output: default via 192.168.1.1 dev eth0 proto dhcp # Default route = gateway of last resort 10.0.0.0/8 via 10.1.1.1 dev eth1 # All 10.x.x.x goes via 10.1.1.1 192.168.1.0/24 dev eth0 proto kernel scope link # Local subnet — direct delivery 172.16.0.0/12 via 192.168.1.254 dev eth0 # RFC 1918 range via specific router

Longest Prefix Match

When multiple routes match a destination, the router picks the most specific (longest subnet mask). Example: a packet to 10.0.1.5:

Route A: 10.0.0.0/8 → matches (covers 10.x.x.x) Route B: 10.0.0.0/16 → matches (covers 10.0.x.x) Route C: 10.0.1.0/24 → matches (covers 10.0.1.x) ← WINS (most specific /24)

Key Routing Protocols

OSPF (Open Shortest Path First)

Link-state protocol. Each router builds a complete map of the network (LSDB). Uses Dijkstra's algorithm to calculate shortest path. Converges fast after topology changes. Used inside enterprise networks and data centres.

BGP (Border Gateway Protocol)

The routing protocol of the internet. Every ISP uses BGP to exchange reachability information. BGP is path-vector — it knows the full AS-path to each destination, not just the next hop. AWS VPC uses BGP for Direct Connect and VPN connections.

# BGP key concepts: AS (Autonomous System) — a network under single admin control, has an ASN eBGP — BGP between different ASes (internet routing) iBGP — BGP within the same AS BGP Route Attributes — AS_PATH, LOCAL_PREF, MED, NEXT_HOP used for path selection BGP Convergence — can take minutes (very slow vs OSPF's seconds)

What is a VPN?

A VPN creates an encrypted tunnel between two endpoints across an untrusted network (the internet). Data is encrypted before leaving the source, travels through the public internet as unintelligible ciphertext, and is decrypted at the destination. From the applications' perspective, they're on the same private network.

IPv6 Essentials

IPv4 addresses ran out in 2011. IPv6 is the future — and increasingly present. You'll encounter it in cloud, ISP, and modern infrastructure roles.

Modern~20 min

Why IPv6?

IPv4 has ~4.3 billion addresses (2³²). With billions of devices, we exhausted them. IPv6 has 2¹²⁸ addresses — ~340 undecillion (340,000,000,000,000,000,000,000,000,000,000,000,000). Every device can have a globally unique address, eliminating the need for NAT.

Network Performance & Metrics

Bandwidth, latency, throughput, jitter — know the differences and how to measure them. Critical for diagnosing slow applications.

Important~20 min

Key Metrics — Defined Precisely

Latency

The time for a single packet to travel from source to destination. Measured in milliseconds (ms). Determined by physical distance (speed of light in fibre ≈ 200,000 km/s) and processing time at each hop. You can't overcome physics — London to Sydney is at minimum ~85ms regardless of your bandwidth.

ping google.com # RTT (Round-Trip Time) = latency × 2 (there and back) # Typical RTTs: same city 1-5ms, same country 10-50ms, cross-continent 80-200ms

Bandwidth

The maximum data transfer rate of a link. Like the width of a pipe. Measured in Mbps or Gbps. A 1 Gbps link can theoretically transfer 125 MB per second. Bandwidth ≠ speed — you can have 1 Gbps bandwidth but 200ms latency.

Throughput

The actual data transfer rate achieved, always ≤ bandwidth. Affected by latency, packet loss, protocol overhead, and TCP window size. A 1 Gbps link with 1% packet loss might only achieve 100 Mbps throughput (TCP slows down aggressively on loss).

Jitter

Variation in packet arrival times. High jitter causes choppy VoIP/video calls even if average latency is fine. Measured in ms. Target: <30ms for VoIP, <10ms for video conferencing.

Packet Loss

Percentage of packets that don't arrive. Even 0.1% loss significantly degrades TCP performance (triggers retransmission + slowdown). Measured with ping or iperf3.

TCP Performance Deep Dive

# TCP Slow Start — how TCP builds up speed Connection starts with small congestion window (cwnd) Each ACK received → double the cwnd (exponential growth) Until cwnd reaches ssthresh (slow-start threshold) Then: linear growth (congestion avoidance) On packet loss: cwnd halved (multiplicative decrease) # This is why large file transfers start slow then accelerate # High latency + high bandwidth = "bandwidth-delay product" problem # Bandwidth-Delay Product (BDP): BDP = Bandwidth × RTT = 1 Gbps × 0.1s (100ms) = 100 Mbit = 12.5 MB # TCP must have 12.5 MB "in flight" to fully utilise this link # Old TCP limit: 64 KB window — only ~5 Mbps on that 1 Gbps link! # TCP window scaling (RFC 7323) solves this for modern systems

MTU — Maximum Transmission Unit

The largest packet size a link can carry without fragmentation. Ethernet MTU = 1500 bytes. If a packet is larger, it must be fragmented (split into multiple packets) or dropped with an ICMP "Fragmentation Needed" message. VPNs reduce the effective MTU (tunnel overhead eats bytes — typically 20-80 bytes). Jumbo frames (MTU 9000) used in data centres for performance.

# Check MTU ip link show eth0 # Shows "mtu 1500" # Test path MTU (find largest unfragmented packet size) ping -M do -s 1472 8.8.8.8 # -M do = don't fragment, 1472+28 header = 1500 # Common MTU issue: VPN tunnel with MTU 1500 → overhead → inner MTU 1420 # Symptom: large files fail/hang but small files/pings work fine

Network Security & Attacks

Know the attacks so you can defend against them. Every infrastructure role requires understanding DDoS, MITM, DNS poisoning, and SYN floods.

Critical~30 min

DDoS — Distributed Denial of Service

Overwhelming a target with traffic from many sources (botnet) until legitimate users can't connect. Different from DoS (single attacker). Attacks happen at different layers:

# Layer 3/4 — Volumetric attacks (flood the pipe) UDP Flood: Millions of UDP packets to random ports → server sends ICMP "unreachable" → overwhelmed ICMP Flood: Massive ping storm Amplification: Send small requests with spoofed source IP to DNS/NTP servers → servers send large responses to victim (amplification 50-100x) # Layer 7 — Application attacks (exhaust server resources) HTTP Flood: Millions of legitimate-looking HTTP GET/POST requests Slowloris: Open many connections, send headers very slowly → holds server threads SSL Exhaustion: TLS handshakes are expensive → flood with partial handshakes # Mitigations: # - AWS Shield (Standard: free, Advanced: $3000/month) # - Cloudflare DDoS protection (absorbs at edge) # - Rate limiting at WAF / nginx level # - Anycast routing (spread attack across many PoPs) # - Blackhole routing (null-route the attack traffic at ISP level)

SYN Flood

Exploits the TCP 3-way handshake. Attacker sends SYN packets with spoofed source IPs. Server allocates memory for each half-open connection waiting for the final ACK that never arrives. Server's connection table fills up — legitimate connections refused.

# Attack: Attacker → SYN (source: fake IPs) Server → SYN-ACK (to fake IPs — no response) Server keeps half-open connections (default timeout 75s) After ~thousands of SYNs: connection table full → all new connections rejected # Mitigation: SYN Cookies (RFC 4987) Server doesn't allocate memory on SYN — instead encodes state in the SYN-ACK sequence number Only allocates memory when valid ACK arrives (proving the client received SYN-ACK) # Enable on Linux: sysctl net.ipv4.tcp_syncookies=1

MITM — Man in the Middle

Attacker intercepts communication between two parties. Both believe they're talking directly to each other.

# ARP Spoofing MITM (on local network): 1. Attacker tells Victim: "The gateway's MAC is MY MAC" 2. Attacker tells Gateway: "The victim's MAC is MY MAC" 3. All Victim ↔ Gateway traffic flows through attacker 4. Attacker reads/modifies unencrypted traffic # SSL Stripping: 1. Victim requests http://bank.com 2. MITM intercepts → fetches https://bank.com (talks HTTPS to server) 3. MITM serves plain http:// to victim 4. Victim's password sent in plaintext! # Mitigations: # - HSTS (HTTP Strict Transport Security): browser always uses HTTPS for domain # - HSTS Preloading: browsers ship with a list of HTTPS-only domains # - Certificate Pinning: app refuses certs not matching known fingerprint # - Dynamic ARP Inspection (DAI): switch validates ARP against DHCP snooping table

DNS Poisoning / Cache Poisoning

Attacker injects false DNS records into a resolver's cache. When users query for example.com, they get the attacker's IP instead — classic phishing attack vector.

# Classic Kaminsky Attack (2008): 1. Attacker floods resolver with queries for random.example.com 2. For each query, sends thousands of fake responses with different transaction IDs 3. One fake response wins the race → poisons the .com TLD cache 4. Now all *.example.com queries return attacker's IP # Mitigations: # DNSSEC — cryptographically signs DNS records # Each response has a digital signature — resolver can verify authenticity # Source port randomisation — makes transaction IDs harder to guess # DNS over HTTPS (DoH) / DNS over TLS (DoT) — encrypt DNS queries dig example.com +dnssec # Check DNSSEC validation

VLANs & Network Segmentation

VLANs divide one physical network into isolated logical networks. Fundamental for security, traffic management, and enterprise networking.

Important~20 min

What is a VLAN?

Virtual LAN. A logical grouping of network devices regardless of their physical location. Devices on the same VLAN can communicate as if on the same switch — devices on different VLANs are isolated even if plugged into the same physical switch. Defined by IEEE 802.1Q standard.

802.1Q VLAN Tagging

A 4-byte tag is inserted into the Ethernet frame header identifying which VLAN the frame belongs to. The tag contains a 12-bit VLAN ID (VID), allowing 4094 VLANs (IDs 1-4094, 0 and 4095 reserved).

Fundamentals

Q: What is the difference between TCP and UDP? When would you use each?

TCP (Transmission Control Protocol) is connection-oriented, reliable, and ordered. It guarantees delivery through acknowledgements and retransmission, ensures packets arrive in order, and has flow control (prevents overwhelming the receiver) and congestion control (prevents overwhelming the network). The cost: overhead of the 3-way handshake, and ACK round-trips add latency.

UDP (User Datagram Protocol) is connectionless, unreliable (fire-and-forget), and has no ordering guarantee. No handshake, no retransmission, no flow control. What you get: low latency and simplicity.

Use TCP for: anything where data integrity matters — HTTP/HTTPS, SSH, FTP, database connections, file transfers. You'd rather wait for a retransmit than receive corrupted data.

Use UDP for: real-time applications where latency matters more than perfection — DNS (fast queries), VoIP (a dropped word is better than a delayed conversation), live video streaming, online gaming, DHCP. HTTP/3 runs over QUIC (a UDP-based protocol) because it implements reliability in user space with better performance than TCP.

Q: Explain the OSI model. What layer does each protocol operate at?

The OSI (Open Systems Interconnection) model is a conceptual framework with 7 layers. Remember: "Please Do Not Throw Sausage Pizza Away" (Physical→Application).

L1 Physical: Bits over wire/fibre/air. Ethernet cables, fibre, Wi-Fi radio waves, hubs (broadcast everything).

L2 Data Link: Frames between devices on same network. MAC addresses, switches (learn which MAC on which port), ARP, 802.1Q VLAN tags, Ethernet.

L3 Network: Packets between networks. IP addresses, routing, routers. IP, ICMP, OSPF, BGP.

L4 Transport: End-to-end connections between processes. Ports, TCP (reliable), UDP (unreliable). Firewalls typically operate here.

L5 Session: Session establishment/management. NetBIOS, RPC, SQL sessions. Often merged with L4/L6 in practice.

L6 Presentation: Data format, encryption, compression. TLS/SSL, JPEG encoding, character set translation (ASCII/Unicode).

L7 Application: Application protocols. HTTP, HTTPS, FTP, SMTP, DNS, DHCP, SSH, WebSocket.

Key interview point: A firewall at L4 sees IP+port but not the URL. A WAF (Web Application Firewall) at L7 sees the full HTTP request and can block based on URL, headers, or body content. A switch operates at L2 (MAC), a router at L3 (IP).

Q: What happens when you ping a machine and get no response?

Systematic diagnosis — work from Layer 1 up:

  1. Is the target machine up? Try pinging its own IP from itself (loopback). If it responds to itself, the machine is running.
  2. Network connectivity: Can you ping the gateway? If not, local network issue — check cables, switch, VLAN.
  3. DNS resolution (if using hostname): Does dig hostname return the right IP? Separate issue from connectivity.
  4. Firewall blocking ICMP: Many servers block ICMP by default. Try curl http://targetip:port — if that works, it's a firewall rule on ICMP, not a connectivity problem. AWS Security Groups don't pass ICMP by default.
  5. Routing problem: Run traceroute — where does the path fail? Asymmetric routing can also cause this (packets go out but return path is broken).
  6. IP conflict: Another device may have the same IP — ARP will send packets to the wrong MAC.

The key insight: no ping response ≠ machine is down. It often just means ICMP is blocked. Always try to connect on the actual service port before concluding the machine is unreachable.

DNS & DHCP

Q: A user says a website is loading slowly. How do you diagnose whether it's a DNS issue?

DNS adds latency to the first connection only (cached after that). To isolate DNS as the cause:

time dig example.com # How long does DNS resolution take? time dig example.com @8.8.8.8 # Try Google's DNS — is your resolver slow? dig example.com +norecurse # Query without recursion — check local cache # If first dig is slow but second is fast → DNS caching working, first lookup was slow # If dig is fast but page is slow → DNS is fine, problem is elsewhere curl -w "%{time_namelookup} %{time_connect} %{time_total}" -o /dev/null https://example.com # time_namelookup = DNS time, time_connect = TCP connect time, time_total = total

If DNS is the issue: switch to a faster resolver (8.8.8.8, 1.1.1.1), check if the authoritative NS is slow, increase TTL so clients cache longer, or use a local caching resolver.

Q: What is TTL in DNS? What happens if you set it too high or too low?

TTL (Time To Live) is the number of seconds a DNS record can be cached before the resolver must refresh it from the authoritative nameserver.

Too high (e.g., 86400 = 24 hours): If you change the IP (failover, migration), users stuck with old IP for up to 24 hours. DNS propagation appears "slow." Most browsers/OS respects TTL strictly.

Too low (e.g., 60 = 1 minute): Every minute every client queries your authoritative nameserver. High query volume, slower lookup for each user (no caching benefit). More nameserver cost.

Best practice: Normal operations: 3600 (1 hour) to 86400 (24 hours). Before a planned change (server migration): lower TTL to 300 (5 min) 24-48 hours before. After the change completes: raise TTL back. This ensures fast propagation of the change while keeping normal operations efficient.

HTTP & TLS

Q: What's the difference between HTTP/1.1, HTTP/2, and HTTP/3?

HTTP/1.1: One request at a time per TCP connection. Workaround: open 6-8 parallel connections per domain. Head-of-line blocking: a slow response blocks everything behind it. Text-based headers (verbose). Introduced keep-alive to reuse connections.

HTTP/2: Multiplexing — multiple requests/responses over a single TCP connection simultaneously. Binary protocol (more compact). Header compression (HPACK). Server push (server can proactively send resources). Still has TCP-level head-of-line blocking — if one TCP packet is lost, all streams wait for retransmission.

HTTP/3: Replaces TCP with QUIC (Quick UDP Internet Connections) — runs over UDP. Each stream is independent — packet loss in one stream doesn't block others (true end of head-of-line blocking). Built-in TLS 1.3 — TLS is part of QUIC, not layered on top. 0-RTT reconnections — returning clients can send data in the first packet. Used by YouTube, Google, Facebook, Cloudflare. ~30% of web traffic is now HTTP/3.

Q: A user gets a "Certificate not trusted" error. Walk me through your diagnosis.

This error means the TLS handshake found a problem with the server's certificate. Possible causes:

  1. Certificate expired: Check expiry with openssl s_client -connect example.com:443 | openssl x509 -noout -dates. Fix: renew the certificate (Let's Encrypt is free with auto-renewal via certbot).
  2. Wrong hostname: The CN/SAN in the cert doesn't match the domain the user typed. A cert for example.com won't work for www.example.com unless it includes both (SAN — Subject Alternative Name). Fix: reissue cert with correct domains.
  3. Self-signed cert: The cert isn't signed by a trusted CA. OK for internal/dev, not for public. Fix: use a public CA (Let's Encrypt, Comodo, DigiCert).
  4. Incomplete certificate chain: The server isn't sending the intermediate CA cert. Browser can't build the chain of trust from the cert to a root CA. Fix: configure the server to include the full chain.
  5. Revoked certificate: The CA revoked it (compromise or mis-issue). Browser checks CRL/OCSP. Fix: get a new cert.
  6. MITM attack: Someone is intercepting traffic and presenting a different cert. The browser catches this. Evidence: sudden unexplained error on a previously working site.
openssl s_client -connect example.com:443 -servername example.com # Look for: Verify return code: 0 (ok) vs error codes # Check: Certificate chain, expiry, issuer, subject curl -v https://example.com 2>&1 | grep -E "SSL|cert|expire|verify"

Routing & Network Design

Q: What is the difference between a hub, switch, and router?

Hub (Layer 1 — Physical): Completely dumb. Receives a signal on one port and broadcasts it out every other port. Every device sees every frame — massive collision domain, huge security risk. Obsolete since mid-2000s.

Switch (Layer 2 — Data Link): Intelligent. Learns MAC address → port mappings by watching traffic. Forwards frames only to the correct port. Each port is its own collision domain. VLANs create separate broadcast domains within one switch. Used for connecting devices within a network (LAN).

Router (Layer 3 — Network): Connects separate networks. Reads destination IP address and forwards packets based on routing table. Creates broadcast domain boundaries (broadcasts don't cross routers). Provides NAT, DHCP, firewall functionality in home/small office routers. Used to connect your LAN to the internet, or VPC subnets to each other.

Layer 3 switch: A switch that also does routing. Can route between VLANs at wire speed without a separate router. Used in enterprise data centres.

Q: You're designing a network for an office. Walk me through your approach.

A structured approach demonstrates experience:

  1. Requirements: How many users? Devices (PCs, phones, printers, IoT, servers)? Bandwidth needs? Remote access needed? Compliance requirements (PCI, HIPAA)?
  2. IP addressing: Choose RFC 1918 range. E.g., 10.0.0.0/8 for flexibility. Plan subnets: 10.0.10.0/24 staff, 10.0.20.0/24 servers, 10.0.30.0/24 management, 10.0.40.0/24 guest.
  3. Segmentation: VLANs for security isolation. Staff PCs never on same VLAN as servers. Guest WiFi completely isolated — VLAN with internet-only access.
  4. Core infrastructure: Managed switches for VLAN support. Layer 3 switch or router for inter-VLAN routing with ACLs. Redundant internet connections (two ISPs) with failover.
  5. Security: Perimeter firewall (pfSense, Fortinet, Cisco ASA). Default-deny inbound. IDS/IPS if needed. 802.1X for wired port authentication.
  6. Remote access: VPN for remote workers. Options: OpenVPN/WireGuard server, or managed solution (Tailscale, Cloudflare Access).
  7. Services: Internal DNS server (resolve internal hostnames). DHCP per VLAN. NTP server. Monitoring (SNMP, syslog to central SIEM).

Cloud & DevOps Networking

Q: An EC2 instance in a private subnet can't reach the internet. What do you check?

Checklist in order:

  1. Route table: Does the private subnet's route table have a route 0.0.0.0/0 → nat-gateway-id? Without this, packets have nowhere to go.
  2. NAT Gateway: Is the NAT Gateway itself in a public subnet? Does the public subnet's route table have 0.0.0.0/0 → igw-xxx (Internet Gateway)?
  3. NAT Gateway state: Is it in "available" state? Not "pending" or "failed"? Check the NAT Gateway's subnet and Elastic IP.
  4. Security Group: Does the EC2 instance's security group allow outbound traffic? (Default SG allows all outbound, but it may have been modified.)
  5. NACL: Does the subnet's Network ACL have outbound rules that allow the traffic? And inbound rules for the return traffic (NACLs are stateless — you need explicit inbound allow for responses).
  6. Test from the instance: SSH in (via bastion or SSM), run curl https://8.8.8.8 (tests IP connectivity) and curl https://google.com (tests DNS too).

Common mistake: forgetting that NACLs are stateless. You must allow both the outbound request AND the inbound response (ephemeral ports 1024-65535). Security groups are stateful — return traffic is automatically allowed.

Q: Explain how Kubernetes networking works (Pod-to-Pod, Service, Ingress).

Pod networking: Every Pod gets a unique IP within the cluster CIDR (e.g., 10.244.0.0/16). Pods can communicate directly with any other Pod by IP without NAT — this is the fundamental Kubernetes networking requirement (implemented by CNI plugins: Calico, Flannel, Cilium, AWS VPC CNI).

Services: Pods are ephemeral — their IPs change. A Service provides a stable virtual IP (ClusterIP) that load-balances to matching pods. kube-proxy on each node programs iptables/IPVS rules to DNAT the ClusterIP to one of the backend Pod IPs. Types: ClusterIP (internal only), NodePort (exposes on each node's port), LoadBalancer (creates cloud LB), ExternalName (DNS alias).

Ingress: A single entry point for HTTP/HTTPS traffic into the cluster. An Ingress Controller (nginx-ingress, AWS ALB Ingress Controller, Traefik) watches Ingress resources and configures a load balancer. Ingress rules define: "host example.com + path /api → backend Service api-service:8080". Handles TLS termination, routing, and virtual hosting.

DNS in Kubernetes: CoreDNS runs in the cluster. Services get DNS names: my-service.my-namespace.svc.cluster.local. Pods get DNS: pod-ip.namespace.pod.cluster.local. Applications use short names (my-service) which resolve via the cluster DNS.

Troubleshooting Scenarios

Q: Users report intermittent connection resets. How do you troubleshoot?

Connection resets (TCP RST) can be caused by several things. Systematic approach:

# 1. Capture the RST to see which side sends it tcpdump -i eth0 'tcp[tcpflags] & tcp-rst != 0' -w reset.pcap # If RST from server → server closed connection (app crash, timeout, max conn exceeded) # If RST from client → client gave up (timeout on client side) # If RST from a middle device → firewall idle timeout (common!) # 2. Check TCP keepalive settings sysctl net.ipv4.tcp_keepalive_time # How long before keepalive probes start (default 7200s) sysctl net.ipv4.tcp_keepalive_intvl # Time between keepalive probes (default 75s) # If a firewall has a 300s idle timeout, connections idle for >5min get RST'd # Fix: lower tcp_keepalive_time to 120s # 3. Check for packet loss mtr --report server_ip # Shows per-hop loss ss -s # Shows retransmit stats # 4. Check application logs for connection pool issues # Database connection pools that hold idle connections get RST after firewall timeout

Most common cause in cloud environments: the load balancer or security group has an idle timeout (AWS ALB default 60 seconds). Long-running idle connections (database connections, WebSockets) get killed. Fix: configure keepalive in your application, or increase the LB idle timeout.

Q: DNS is working but the website still shows the old IP after you changed the DNS record. Why?

This is a DNS caching problem. Multiple layers cache DNS records:

  1. Browser cache: Chrome caches DNS internally. Clear it: chrome://net-internals/#dns → Clear host cache. Or open a new incognito window.
  2. OS cache: The OS DNS resolver caches results. On Linux: sudo systemd-resolve --flush-caches. On Windows: ipconfig /flushdns.
  3. ISP/upstream resolver cache: Your DNS resolver (ISP or 8.8.8.8) cached the old record. Must wait for its TTL to expire — you can't flush someone else's cache remotely.
  4. The old TTL is still active: The TTL was set before you changed the record. Resolvers must wait that many seconds before re-querying. If TTL was 86400 (24h), it could take 24 hours.

Test what IP you're getting: dig example.com @8.8.8.8 vs dig example.com @1.1.1.1. If different resolvers return different IPs, propagation is in progress — some resolvers have the old cached value, some have already refreshed.

Prevention: Always lower TTL to 300 at least 24 hours before making a DNS change. After the change is confirmed working, raise TTL back.
