CI/CD Concepts

Continuous Integration, Delivery, and Deployment — what they actually mean.

Foundation~20 min

CI vs CD vs CD

TermWhat It MeansAutomated?
Continuous Integration (CI)Every code push triggers: build + lint + test. Fast feedback on broken code.Fully automated
Continuous Delivery (CD)Every passing build is packaged and ready to deploy to production. A human presses deploy.Auto up to staging; manual prod gate
Continuous Deployment (CD)Every passing build automatically deploys to production. No human gate.Fully automated end-to-end

The Pipeline Flow

Why CI/CD Matters

Without CI/CDWith CI/CD
Manual deployments → human errorAutomated, consistent every time
Deploy monthly → huge risky releasesDeploy daily/hourly → small safe changes
Hours to find broken codeBroken code caught in minutes
"Works on my machine"Tested in identical environment
Fear of deployingConfidence to deploy anytime

DORA Metrics — What Companies Measure
Elite companies (Google, Netflix, Amazon) achieve:
• Deployment frequency: multiple times per day
• Lead time for changes: less than 1 hour (code → production)
• Change failure rate: less than 5%
• MTTR (recovery time): less than 1 hour

GitHub Actions — Full Pipeline

Every component of a production GitHub Actions workflow explained.

Industry Standard~30 min

Workflow Anatomy

yaml.github/workflows/ci.yml — fully annotated
# Workflow name — shown in GitHub Actions UI
name: CI Pipeline

# TRIGGERS — when does this run?
on:
  push:
    branches: [main, develop]  # run on push to these branches
  pull_request:
    branches: [main]            # run on PRs targeting main
  workflow_dispatch:              # allow manual trigger from UI

# ENV vars available to all jobs
env:
  AWS_REGION: us-east-1
  ECR_REPOSITORY: myapp

jobs:
  # ===== JOB 1: Test =====
  test:
    name: Run Tests
    runs-on: ubuntu-latest    # GitHub-hosted runner (free)
    steps:
      - name: Checkout code
        uses: actions/checkout@v4   # clones your repo

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'             # cache node_modules between runs

      - name: Install dependencies
        run: npm ci

      - name: Run linting
        run: npm run lint

      - name: Run unit tests
        run: npm test -- --coverage

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

  # ===== JOB 2: Build & Push to ECR =====
  build:
    name: Build & Push Image
    runs-on: ubuntu-latest
    needs: test                  # only runs if test job passes
    if: github.ref == 'refs/heads/main'  # only on main branch
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}  # OIDC — no static keys!
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Set image tag
        id: meta
        run: |
          echo "IMAGE_TAG=${GITHUB_SHA::8}" >> $GITHUB_ENV
          echo "ECR_REGISTRY=${{ steps.login-ecr.outputs.registry }}" >> $GITHUB_ENV

      - name: Build and push Docker image
        run: |
          docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
          docker tag $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG $ECR_REGISTRY/$ECR_REPOSITORY:latest
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest
          echo "image=$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG" >> $GITHUB_OUTPUT

Full Pipeline: GitHub Actions → ECR → EKS

Complete production pipeline. Every push to main deploys to your EKS cluster.

AWS Production Industry Standard

Step 1: Set Up OIDC (No Static AWS Keys)

The production way — GitHub Actions assumes an IAM role via OIDC. Never store AWS_ACCESS_KEY_ID as a secret.

bashCreate OIDC provider and IAM role
# Create OIDC identity provider in AWS IAM
aws iam create-open-id-connect-provider \
  --url https://token.actions.githubusercontent.com \
  --client-id-list sts.amazonaws.com \
  --thumbprint-list 6938fd4d98bab03faadb97b34396831e3780aea1

# Create trust policy (replace YOUR_GITHUB_ORG/REPO)
cat > trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::123456789:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:YOUR_GITHUB_ORG/REPO:*"
      }
    }
  }]
}
EOF

# Create IAM role
aws iam create-role \
  --role-name github-actions-eks-role \
  --assume-role-policy-document file://trust-policy.json

# Attach policies
aws iam attach-role-policy \
  --role-name github-actions-eks-role \
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser

# Add the role ARN to GitHub repo secrets as AWS_ROLE_ARN

Step 2: Grant GitHub Actions Access to EKS

bash
# Get current aws-auth ConfigMap
kubectl get configmap aws-auth -n kube-system -o yaml

# Add the GitHub Actions role to aws-auth
eksctl create iamidentitymapping \
  --cluster prod-cluster \
  --region us-east-1 \
  --arn arn:aws:iam::123456789:role/github-actions-eks-role \
  --group system:masters \
  --username github-actions

Step 3: The Complete Workflow

yaml.github/workflows/deploy.yml — production pipeline
name: Build, Test and Deploy to EKS

on:
  push:
    branches: [main]
  workflow_dispatch:

env:
  AWS_REGION: us-east-1
  EKS_CLUSTER: prod-cluster
  ECR_REPOSITORY: myapp
  NAMESPACE: production
  DEPLOYMENT_NAME: myapp-deployment

jobs:
  test:
    name: 🧪 Test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '18', cache: 'npm' }
      - run: npm ci
      - run: npm run lint
      - run: npm test

  build-and-push:
    name: 🐳 Build & Push to ECR
    runs-on: ubuntu-latest
    needs: test
    outputs:
      image-tag: ${{ steps.build.outputs.image-tag }}
      ecr-registry: ${{ steps.login-ecr.outputs.registry }}
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build, tag, push image
        id: build
        env:
          REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          IMAGE_TAG: ${{ github.sha }}
        run: |
          docker build \
            --build-arg APP_VERSION=${{ github.sha }} \
            -t $REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG \
            -t $REGISTRY/$ECR_REPOSITORY:latest \
            .
          docker push $REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
          docker push $REGISTRY/$ECR_REPOSITORY:latest
          echo "image-tag=$IMAGE_TAG" >> $GITHUB_OUTPUT

  deploy:
    name: 🚀 Deploy to EKS
    runs-on: ubuntu-latest
    needs: build-and-push
    environment: production   # requires manual approval if configured
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Update kubeconfig
        run: |
          aws eks update-kubeconfig \
            --name $EKS_CLUSTER \
            --region $AWS_REGION

      - name: Deploy to EKS (rolling update)
        env:
          REGISTRY: ${{ needs.build-and-push.outputs.ecr-registry }}
          IMAGE_TAG: ${{ needs.build-and-push.outputs.image-tag }}
        run: |
          kubectl set image deployment/$DEPLOYMENT_NAME \
            myapp=$REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG \
            --namespace $NAMESPACE

          kubectl rollout status deployment/$DEPLOYMENT_NAME \
            --namespace $NAMESPACE \
            --timeout=5m

      - name: Verify deployment
        run: |
          kubectl get pods -n $NAMESPACE
          kubectl get ingress -n $NAMESPACE

      - name: Notify on failure (rollback)
        if: failure()
        run: |
          echo "Deployment failed! Rolling back..."
          kubectl rollout undo deployment/$DEPLOYMENT_NAME -n $NAMESPACE
          kubectl rollout status deployment/$DEPLOYMENT_NAME -n $NAMESPACE

Jenkins Pipeline (Jenkinsfile)

Declarative Jenkinsfile for Docker → ECR → EKS. Still widely used in enterprises.

Enterprise StandardInterview Common

Declarative Jenkinsfile

groovyJenkinsfile
pipeline {
    agent {
        kubernetes {            // run pipeline inside K8s pod
            yaml """
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: docker
    image: docker:24-dind
    securityContext:
      privileged: true
  - name: kubectl
    image: bitnami/kubectl:latest
    command: ['sleep', 'infinity']
"""
        }
    }

    environment {
        AWS_REGION          = 'us-east-1'
        ECR_REGISTRY        = "123456789.dkr.ecr.us-east-1.amazonaws.com"
        ECR_REPOSITORY      = 'myapp'
        EKS_CLUSTER         = 'prod-cluster'
        IMAGE_TAG           = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
        AWS_CREDENTIALS_ID  = 'aws-credentials'  // Jenkins credential ID
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
                sh 'echo "Building commit: ${IMAGE_TAG}"'
            }
        }

        stage('Test') {
            steps {
                sh '''
                    npm ci
                    npm run lint
                    npm test
                '''
            }
            post {
                always {
                    junit 'test-results/*.xml'
                }
            }
        }

        stage('Build Docker Image') {
            steps {
                container('docker') {
                    sh """
                        docker build \
                          -t ${ECR_REGISTRY}/${ECR_REPOSITORY}:${IMAGE_TAG} \
                          -t ${ECR_REGISTRY}/${ECR_REPOSITORY}:latest \
                          .
                    """
                }
            }
        }

        stage('Push to ECR') {
            steps {
                container('docker') {
                    withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                                       credentialsId: AWS_CREDENTIALS_ID]]) {
                        sh """
                            aws ecr get-login-password --region ${AWS_REGION} | \
                              docker login --username AWS --password-stdin ${ECR_REGISTRY}
                            docker push ${ECR_REGISTRY}/${ECR_REPOSITORY}:${IMAGE_TAG}
                            docker push ${ECR_REGISTRY}/${ECR_REPOSITORY}:latest
                        """
                    }
                }
            }
        }

        stage('Deploy to Staging') {
            steps {
                container('kubectl') {
                    withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                                       credentialsId: AWS_CREDENTIALS_ID]]) {
                        sh """
                            aws eks update-kubeconfig --name ${EKS_CLUSTER} --region ${AWS_REGION}
                            kubectl set image deployment/myapp-deployment \
                              myapp=${ECR_REGISTRY}/${ECR_REPOSITORY}:${IMAGE_TAG} \
                              -n staging
                            kubectl rollout status deployment/myapp-deployment -n staging --timeout=5m
                        """
                    }
                }
            }
        }

        stage('Approve Production') {
            when { branch 'main' }
            steps {
                timeout(time: 24, unit: 'HOURS') {
                    input message: "Deploy ${IMAGE_TAG} to Production?",
                          ok: 'Deploy Now'
                }
            }
        }

        stage('Deploy to Production') {
            when { branch 'main' }
            steps {
                container('kubectl') {
                    withCredentials([[$class: 'AmazonWebServicesCredentialsBinding',
                                       credentialsId: AWS_CREDENTIALS_ID]]) {
                        sh """
                            kubectl set image deployment/myapp-deployment \
                              myapp=${ECR_REGISTRY}/${ECR_REPOSITORY}:${IMAGE_TAG} \
                              -n production
                            kubectl rollout status deployment/myapp-deployment -n production --timeout=10m
                        """
                    }
                }
            }
        }
    }

    post {
        success {
            slackSend(channel: '#deployments',
                       color: 'good',
                       message: "✅ Deployed ${IMAGE_TAG} to production")
        }
        failure {
            slackSend(channel: '#deployments',
                       color: 'danger',
                       message: "❌ Pipeline FAILED for ${IMAGE_TAG}")
            container('kubectl') {
                sh 'kubectl rollout undo deployment/myapp-deployment -n production || true'
            }
        }
    }
}

ArgoCD — GitOps

Git is the source of truth. ArgoCD syncs your cluster to match your repo automatically.

GitOps PatternInterview Hot Topic

What is GitOps?

In GitOps, the desired state of your infrastructure is stored in Git. ArgoCD continuously watches the repo and syncs the cluster to match it. No more "kubectl apply" in pipelines — just git push.

Install ArgoCD on EKS

bash
# Install ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Wait for pods
kubectl wait --for=condition=ready pod \
  -l app.kubernetes.io/name=argocd-server \
  -n argocd --timeout=300s

# Expose ArgoCD UI (LoadBalancer)
kubectl patch svc argocd-server -n argocd \
  -p '{"spec": {"type": "LoadBalancer"}}'

# Get initial admin password
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# Get ArgoCD URL
kubectl get svc argocd-server -n argocd \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'

# Login via CLI
argocd login <ARGOCD_URL> --username admin --password <PASSWORD>

Create an ArgoCD Application

yamlargocd-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/myapp-k8s-config
    targetRevision: HEAD
    path: k8s/production         # folder with K8s YAML files
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true             # delete K8s resources removed from Git
      selfHeal: true          # revert manual kubectl changes
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 5
      backoff:
        duration: 5s
        maxDuration: 3m
        factor: 2

GitOps Workflow with CI/CD + ArgoCD

yaml.github/workflows/gitops.yml — update image tag in Git
name: GitOps Deploy
on:
  push:
    branches: [main]

jobs:
  build-and-update:
    runs-on: ubuntu-latest
    steps:
      # ... test, build, push to ECR steps ...

      - name: Update image tag in k8s-config repo
        run: |
          # Clone the separate K8s config repo
          git clone https://x-access-token:${{ secrets.CONFIG_REPO_TOKEN }}@github.com/myorg/myapp-k8s-config
          cd myapp-k8s-config

          # Update the image tag in the deployment YAML
          sed -i "s|image: .*myapp:.*|image: ${ECR_REGISTRY}/myapp:${GITHUB_SHA}|" \
            k8s/production/deployment.yaml

          # Commit and push — ArgoCD detects this and deploys!
          git config user.email "github-actions@github.com"
          git config user.name "GitHub Actions"
          git add -A
          git commit -m "chore: update myapp image to ${GITHUB_SHA}"
          git push
🎯
GitOps vs Traditional CD — Interview Answer
Traditional CD: CI pipeline runs kubectl apply directly against the cluster. It needs cluster credentials in CI, and manual kubectl changes aren't tracked.

GitOps (ArgoCD): Git is the single source of truth. CI only updates Git. ArgoCD pulls and syncs. Benefits: audit trail, easy rollback (git revert), no cluster credentials in CI, self-healing (reverts manual changes).

Blue-Green & Canary Deployments

Zero-downtime deployment strategies used at Netflix, Amazon, Google.

Interview FavoriteProduction Pattern

Deployment Strategy Comparison

StrategyHow It WorksDowntimeRollback SpeedResource Cost
RecreateKill all v1 pods, start all v2 podsYesSlow1x
Rolling UpdateReplace pods one at a timeNoMedium (rollout undo)1x + surge
Blue-GreenRun v1 and v2 simultaneously, switch traffic instantlyNoInstant (switch back)2x
CanaryRoute 5-10% traffic to v2, monitor, then increaseNoFast (remove canary)1.1x

Blue-Green on Kubernetes

yamlblue-green deployment
# BLUE deployment (current production — v1)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
spec:
  replicas: 3
  selector:
    matchLabels: { app: myapp, slot: blue }
  template:
    metadata:
      labels: { app: myapp, slot: blue }
    spec:
      containers:
        - image: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0.0
---
# GREEN deployment (new version — v2)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
spec:
  replicas: 3
  selector:
    matchLabels: { app: myapp, slot: green }
  template:
    metadata:
      labels: { app: myapp, slot: green }
    spec:
      containers:
        - image: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v2.0.0
---
# SERVICE — points to blue OR green via selector
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  selector:
    app: myapp
    slot: blue   # ← switch this to "green" to deploy!
  ports:
    - { port: 80, targetPort: 3000 }
bashSwitch traffic blue → green (instant, zero downtime)
# Deploy green (v2) alongside blue (v1)
kubectl apply -f green-deployment.yaml

# Test green internally
kubectl run test --image=curlimages/curl --rm -it --restart=Never \
  -- curl http://myapp-green-service/health

# Switch ALL traffic from blue to green (atomic, no downtime)
kubectl patch service myapp-service \
  -p '{"spec":{"selector":{"slot":"green"}}}'

# Verify
kubectl get service myapp-service -o yaml | grep slot

# Rollback = switch back to blue (instant!)
kubectl patch service myapp-service \
  -p '{"spec":{"selector":{"slot":"blue"}}}'

# Clean up old blue deployment when satisfied
kubectl delete deployment myapp-blue

Canary Deployment

yamlcanary deployment — 10% traffic to v2
# Main deployment: 9 replicas of v1 apiVersion: apps/v1 kind: Deployment metadata: name: myapp-stable spec: replicas: 9 # 9 out of 10 pods selector: matchLabels: { app: myapp } template: metadata: labels: { app: myapp, track: stable } spec: containers: - image: myapp:v1.0.0 --- # Canary: 1 replica of v2 (10% of traffic) apiVersion: apps/v1 kind: Deployment metadata: name: myapp-canary spec: replicas: 1 # 1 out of 10 pods = 10% traffic selector: matchLabels: { app: myapp } template: metadata: labels: { app: myapp, track: canary } spec: containers: - image: myapp:v2.0.0 --- # ONE service routes to BOTH (by matching app: myapp label) apiVersion: v1 kind: Service metadata: name: myapp-service spec: selector: app: myapp # matches BOTH stable and canary pods
