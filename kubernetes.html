What is Kubernetes?

Why Docker alone isn't enough for production — and what K8s solves.

Beginner~20 min

The Problem Kubernetes Solves

Docker runs containers. But in production you need answers to:

  • What happens when a container crashes? Who restarts it?
  • How do you scale from 3 to 30 containers under load?
  • How does traffic get load-balanced across 30 replicas?
  • How do you update your app with zero downtime?
  • How do you roll back a bad deployment in 10 seconds?
  • How do you spread containers across multiple servers?

Kubernetes is the answer to all of these. It's a container orchestration platform — it manages containers across a cluster of machines.

🎯
Interview definition
"Kubernetes is an open-source container orchestration system that automates deployment, scaling, self-healing, and management of containerized applications across a cluster of nodes. It was open-sourced by Google in 2014, based on their internal Borg system."

What K8s Does Automatically

FeatureWhat K8s Does
Self-healingRestarts crashed containers, replaces failed nodes
Auto-scalingAdds/removes pods based on CPU/memory (HPA)
Rolling updatesUpdates pods one at a time — zero downtime
Load balancingDistributes traffic across all healthy pods
Service discoveryBuilt-in DNS — pods find each other by name
Secret managementInject credentials without hardcoding
Storage orchestrationAuto-provision EBS volumes for stateful apps
Node managementSchedules pods onto nodes with available resources

Kubernetes Architecture

Control Plane vs Worker Nodes — know every component cold for interviews.

Interview Must-Know~30 min

The Two Sides of a K8s Cluster

Every Component Explained

Control Plane Components

ComponentRoleInterview Tip
kube-apiserverThe front door — all kubectl/SDK calls hit this REST API. Validates and stores state in etcd.Everything goes through the API server. It's stateless.
etcdDistributed key-value store. The source of truth for all cluster state.If etcd goes down, the cluster still runs but can't accept changes. Always run 3 or 5 nodes.
kube-schedulerWatches for unscheduled pods and assigns them to nodes based on resources, affinity rules, taints.Doesn't run pods — only decides WHERE they go.
kube-controller-managerRuns control loops: ReplicaSet (maintains desired count), Node controller (handles failed nodes), Job controller.The reconciliation loop: "desired state vs actual state"
cloud-controller-managerAWS-specific: creates Load Balancers, EBS volumes, Security Groups when K8s resources are created.This is why type: LoadBalancer creates an AWS ELB automatically.

Worker Node Components

ComponentRole
kubeletAgent on every node. Receives pod specs from API server and tells the container runtime to start/stop containers. Reports node status back.
kube-proxyMaintains iptables/IPVS rules for Service networking. Routes traffic to the correct pod endpoints.
Container RuntimeActually runs containers (containerd, CRI-O). Kubernetes deprecated direct Docker support in v1.24 — uses containerd directly via CRI.
🎯
What Happens When You Run kubectl apply -f deployment.yaml
1. kubectl → sends YAML to kube-apiserver (REST POST)
2. apiserver validates → writes to etcd
3. ReplicaSet controller sees new pods needed → creates Pod objects
4. Scheduler assigns each Pod to a Node
5. kubelet on that Node sees new Pod → tells containerd to pull image & start container
6. Pod is Running ✓

Create an EKS Cluster on AWS

Real AWS. Real cluster. The exact steps used by production teams.

AWS Real CloudHands-On
☁️
Cost Note
EKS control plane costs ~$0.10/hour (~$2.40/day). 3× t3.medium nodes cost ~$0.13/hour each. Total: ~$0.50/hour. Delete the cluster when done to avoid charges. The full lab runs in under 2 hours.

Step 1: Install Required Tools

bashInstall kubectl, eksctl, and AWS CLI
# Install kubectl
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
chmod +x kubectl && sudo mv kubectl /usr/local/bin/
kubectl version --client

# Install eksctl (EKS cluster management tool)
curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin
eksctl version

# Install Helm (K8s package manager)
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
helm version

# Configure AWS CLI
aws configure
# Enter your Access Key ID, Secret, Region: us-east-1

Step 2: Create the EKS Cluster

bash
# Create cluster (takes ~15 minutes)
eksctl create cluster \
  --name prod-cluster \
  --region us-east-1 \
  --nodegroup-name standard-workers \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 5 \
  --managed                   # managed node group (AWS handles node OS updates)

# eksctl creates:
#  - VPC with public/private subnets
#  - EKS control plane
#  - EC2 node group (3× t3.medium)
#  - IAM roles and security groups
#  - Updates your ~/.kube/config automatically

Step 3: Verify & Connect

bash
# Update kubeconfig (if not done by eksctl)
aws eks update-kubeconfig --name prod-cluster --region us-east-1

# Verify connection
kubectl get nodes
kubectl get nodes -o wide   # shows EC2 instance IDs and IPs

# Check cluster info
kubectl cluster-info
kubectl get pods -A          # see kube-system pods

Step 4: Install AWS Load Balancer Controller

Required for Ingress with AWS ALB. Every production EKS cluster needs this.

bash
# Create IAM OIDC provider eksctl utils associate-iam-oidc-provider \ --region us-east-1 \ --cluster prod-cluster \ --approve # Create IAM policy for ALB controller curl -O https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.6.0/docs/install/iam_policy.json aws iam create-policy \ --policy-name AWSLoadBalancerControllerIAMPolicy \ --policy-document file://iam_policy.json # Create service account eksctl create iamserviceaccount \ --cluster prod-cluster \ --namespace kube-system \ --name aws-load-balancer-controller \ --attach-policy-arn arn:aws:iam::${AWS_ACCOUNT_ID}:policy/AWSLoadBalancerControllerIAMPolicy \ --override-existing-serviceaccounts \ --approve # Install via Helm helm repo add eks https://aws.github.io/eks-charts helm install aws-load-balancer-controller eks/aws-load-balancer-controller \ --namespace kube-system \ --set clusterName=prod-cluster \ --set serviceAccount.create=false \ --set serviceAccount.name=aws-load-balancer-controller

Pods — The Smallest Unit

What a pod is, why you almost never create them directly, and when you do.

Core ConceptHands-On

What is a Pod?

A Pod is a wrapper around one or more containers that share network and storage. Every container in a pod shares the same IP address and can communicate via localhost.

💡
99% of the time: one container per pod
Multiple containers in a pod are for sidecar patterns — e.g., an app container + a logging sidecar. For interviews: "A pod can have multiple containers that share the same network namespace and storage volumes, but in practice most pods run a single container."

Pod YAML — Every Field Explained

yamlpod.yaml — complete with all important fields
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  namespace: default
  labels:                    # key-value tags — Services use these for routing
    app: myapp
    version: v1
    env: production
spec:
  containers:
    - name: myapp
      image: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0.0
      ports:
        - containerPort: 3000
      env:
        - name: NODE_ENV
          value: "production"
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:           # pull from K8s Secret
              name: myapp-secret
              key: db-password
      resources:
        requests:               # minimum guaranteed (used for scheduling)
          cpu: "100m"            # 100 millicores = 0.1 CPU core
          memory: "128Mi"
        limits:                 # maximum allowed (OOMKilled if exceeded)
          cpu: "500m"
          memory: "512Mi"
      livenessProbe:           # K8s restarts container if this fails
        httpGet:
          path: /health
          port: 3000
        initialDelaySeconds: 15
        periodSeconds: 10
        failureThreshold: 3
      readinessProbe:          # K8s removes from Service if this fails
        httpGet:
          path: /ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 5
  restartPolicy: Always      # Always | OnFailure | Never
⚠️
Liveness vs Readiness Probe — Interview Question
Liveness: "Is the container still alive?" — if it fails, K8s restarts the container.
Readiness: "Is the container ready to serve traffic?" — if it fails, K8s removes it from the Service (stops sending traffic) but doesn't restart.

Use readiness for: app startup time, dependency checks. Use liveness for: deadlock detection, memory leaks.

Deployments & ReplicaSets

How K8s maintains desired state and enables zero-downtime updates.

Interview Must-KnowHands-On

The Deployment Hierarchy

Deployment → manages → ReplicaSet → manages → Pods

You create a Deployment. It creates a ReplicaSet. The ReplicaSet creates and maintains the correct number of Pods. On update, Deployment creates a new ReplicaSet, gradually shifting pods over.

Complete Production Deployment YAML

yamldeployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-deployment
  namespace: production
  labels:
    app: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp              # must match template.metadata.labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # allow 1 extra pod during update (4 total)
      maxUnavailable: 0    # never go below 3 available (zero-downtime)
  template:
    metadata:
      labels:
        app: myapp
        version: v1.2.0
    spec:
      containers:
        - name: myapp
          image: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.0
          ports:
            - containerPort: 3000
          env:
            - name: NODE_ENV
              value: "production"
          envFrom:
            - configMapRef:          # load all keys from ConfigMap as env vars
                name: myapp-config
            - secretRef:
                name: myapp-secret
          resources:
            requests: { cpu: "100m", memory: "256Mi" }
            limits:   { cpu: "500m", memory: "512Mi" }
          livenessProbe:
            httpGet: { path: /health, port: 3000 }
            initialDelaySeconds: 15
            periodSeconds: 10
          readinessProbe:
            httpGet: { path: /ready, port: 3000 }
            initialDelaySeconds: 5
            periodSeconds: 5
      affinity:
        podAntiAffinity:          # spread pods across nodes
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchExpressions:
                    - key: app
                      operator: In
                      values: [myapp]
                topologyKey: kubernetes.io/hostname

Services — Network Access to Pods

ClusterIP, NodePort, LoadBalancer. When to use each.

Interview Must-KnowHands-On

Why Services?

Pods are ephemeral — they get new IPs when restarted. A Service gives you a stable IP and DNS name that always routes to healthy pods matching its selector.

Service TypeAccessible FromAWS CreatesUse Case
ClusterIPInside cluster onlyNothingInternal service-to-service communication
NodePortAny node IP + portNothingDevelopment/testing only
LoadBalancerInternetNLB/ELB automaticallyExpose service to internet (TCP/UDP)
ExternalNameInside clusterNothingMap to external DNS (RDS, ElastiCache)

Service YAML Examples

yamlservice-clusterip.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
  namespace: production
spec:
  type: ClusterIP          # default — internal only
  selector:
    app: myapp              # routes to pods with this label
  ports:
    - port: 80              # port other services use to reach this
      targetPort: 3000     # port on the pod
      protocol: TCP

# Other services reach myapp at: http://myapp-service.production.svc.cluster.local
# or just: http://myapp-service  (within same namespace)
yamlservice-loadbalancer.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-lb
  annotations:
    # AWS NLB (Network Load Balancer) — Layer 4, faster
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
    service.beta.kubernetes.io/aws-load-balancer-cross-zone-load-balancing-enabled: "true"
spec:
  type: LoadBalancer
  selector:
    app: myapp
  ports:
    - port: 80
      targetPort: 3000

# AWS creates NLB automatically!
# kubectl get services → shows EXTERNAL-IP = abc123.elb.amazonaws.com
yamlservice-externalname.yaml — map RDS to K8s DNS
apiVersion: v1
kind: Service
metadata:
  name: postgres           # pods use this as hostname
  namespace: production
spec:
  type: ExternalName
  externalName: mydb.abc123.us-east-1.rds.amazonaws.com

# Now pods can connect to postgres:5432
# instead of hardcoding the long RDS hostname!

Secret — Sensitive Data

bashCreate secrets (never commit to git!)
# Create from literal values (base64 encoded automatically)
kubectl create secret generic myapp-secret \
  --from-literal=db-password=supersecretpassword \
  --from-literal=jwt-secret=myjwtsecretkey \
  --from-literal=redis-password=redispass \
  --namespace production

# Create from file
kubectl create secret generic tls-cert \
  --from-file=tls.crt=server.crt \
  --from-file=tls.key=server.key

# Create TLS secret type
kubectl create secret tls myapp-tls \
  --cert=server.crt \
  --key=server.key
yamlsecret.yaml (if you must write it — encode values!)
apiVersion: v1
kind: Secret
metadata:
  name: myapp-secret
type: Opaque
data:
  # Values MUST be base64 encoded: echo -n "value" | base64
  db-password: c3VwZXJzZWNyZXRwYXNzd29yZA==
  jwt-secret: bXlqd3RzZWNyZXRrZXk=
⚠️
K8s Secrets are NOT encrypted by default
Kubernetes Secrets are only base64-encoded (not encrypted) in etcd by default. For production on AWS: enable etcd encryption at rest on EKS, and better yet use AWS Secrets Manager with the Secrets Store CSI Driver or External Secrets Operator. Interviewers love this distinction.

AWS Secrets Manager Integration (Production Pattern)

bash
# Store secret in AWS Secrets Manager aws secretsmanager create-secret \ --name production/myapp/db-password \ --secret-string '{"password":"supersecret","username":"admin"}' # Install External Secrets Operator helm repo add external-secrets https://charts.external-secrets.io helm install external-secrets external-secrets/external-secrets -n external-secrets --create-namespace
yamlexternalsecret.yaml

kind: ExternalSecret

  name: myapp-secret
spec:
  refreshInterval: 1h

Ingress & AWS ALB

Route HTTP/HTTPS traffic to multiple services. TLS termination on AWS.

AWS ProductionInterview Favorite

Why Ingress?

Instead of one LoadBalancer per service (expensive — each creates an AWS ELB), Ingress uses ONE load balancer and routes traffic to multiple services based on hostname or URL path.

yamlingress.yaml — AWS ALB with TLS
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  namespace: production
  annotations:
    # Use AWS ALB (Application Load Balancer)
    kubernetes.io/ingress.class: "alb"
    alb.ingress.kubernetes.io/scheme: "internet-facing"
    alb.ingress.kubernetes.io/target-type: "ip"
    # TLS with ACM certificate
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTP": 80}, {"HTTPS": 443}]'
    alb.ingress.kubernetes.io/certificate-arn: "arn:aws:acm:us-east-1:123456789:certificate/abc-123"
    alb.ingress.kubernetes.io/ssl-redirect: "443"
    # Health check
    alb.ingress.kubernetes.io/healthcheck-path: "/health"
    # Sticky sessions
    alb.ingress.kubernetes.io/target-group-attributes: "stickiness.enabled=true,stickiness.lb_cookie.duration_seconds=86400"
spec:
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-service
                port: { number: 80 }
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port: { number: 80 }
    - host: admin.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: admin-service
                port: { number: 80 }

HPA — Horizontal Pod Autoscaler

Automatically scale pods based on CPU, memory, or custom metrics.

Production CriticalInterview Favorite

HPA YAML

yamlhpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp-deployment
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # scale up if CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: AverageValue
          averageValue: 400Mi
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30    # fast scale up
    scaleDown:
      stabilizationWindowSeconds: 300   # slow scale down (5 min)

Test rolling update and rollback
›
bash
# Update image (simulates deploying new version)
kubectl set image deployment/myapp-deployment \
  myapp=123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.1.0 \
  --namespace production

# Watch the rolling update
kubectl rollout status deployment/myapp-deployment -n production

# Check history
kubectl rollout history deployment/myapp-deployment -n production

# Rollback if needed
kubectl rollout undo deployment/myapp-deployment -n production
kubectl rollout status deployment/myapp-deployment -n production
