🐳 Docker Questions
1
What is the difference between a Docker image and a Docker container?
Easy ›

Image: A read-only template — a snapshot of the application and all its dependencies, built layer by layer from a Dockerfile. Stored in a registry. Like a class in OOP.

Container: A running instance of an image. Has a writable layer on top. Multiple containers can run from the same image simultaneously. Like an object (instance) of a class.

Key point: Images are immutable. When a container writes data, it writes to its own writable layer — the original image is unchanged. This is why you need volumes for persistent data.

bash
docker images          # list images (templates)
docker ps              # list running containers (instances)
docker ps -a           # all containers including stopped
2
What is a Dockerfile and what are the most important instructions?
Easy ›

A Dockerfile is a text file with instructions to build a Docker image. Each instruction creates a layer.

Most important instructions:

  • FROM — base image (always first)
  • WORKDIR — set working directory
  • COPY — copy files from host to image
  • RUN — execute commands during build (creates layers)
  • ENV — set environment variables
  • EXPOSE — document which port the app uses
  • CMD — default command to run (can be overridden)
  • ENTRYPOINT — main process, cannot be overridden
  • USER — run as non-root user

Interview tip: Explain the difference between CMD and ENTRYPOINT. CMD provides defaults that can be overridden. ENTRYPOINT is the main executable. Together: ENTRYPOINT=["node"] CMD=["app.js"] → runs "node app.js", but you can override CMD with: docker run img server.js

3
Explain Docker layer caching and how to optimize Dockerfiles for it.
Medium ›

Docker caches each layer based on its instruction + the content it depends on. If neither changes, the cached layer is reused.

The optimization trick: Put instructions that change frequently (your source code) AFTER instructions that change rarely (dependencies).

dockerfile
# BAD — code change invalidates npm install cache
COPY . .
RUN npm install    # re-runs on EVERY code change!

# GOOD — separate deps from code
COPY package*.json ./
RUN npm ci         # cached unless package.json changes
COPY . .           # only this layer re-runs on code change

Result: builds go from 2 minutes to 5 seconds because npm install is cached. This is one of the highest-impact Docker optimizations in practice.

4
What is a multi-stage build and why use it?
Medium ›

A multi-stage build uses multiple FROM statements. Each stage can use a different base image. The final image only contains what you explicitly COPY from previous stages.

Why: Build tools (npm, webpack, compilers) needed to build the app are NOT needed to run it. Multi-stage lets you build in a heavy image and ship a minimal one.

dockerfile
# Build stage — has full Go toolchain (~1GB)
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN go build -o /myapp .

# Final stage — just the binary (~10MB)
FROM scratch
COPY --from=builder /myapp /myapp
ENTRYPOINT ["/myapp"]

# Image goes from 1GB → ~10MB (99% smaller)

Real impact: Smaller images = faster pull times, smaller attack surface, lower ECR storage costs, faster K8s pod startup.

5
How does Docker networking work? Explain bridge, host, and overlay networks.
Medium ›

Bridge (default): Creates a virtual network interface (docker0). Containers get their own IP (172.17.0.x). Communicate with each other by IP. On user-defined bridge networks, Docker provides DNS — containers communicate by name. Port mapping (-p) is required for host access.

Host: Container shares the host's network namespace directly. No network isolation. Container listens on host ports without -p mapping. Best performance but worst isolation. Used for performance-critical apps.

None: No network interface. Complete isolation. For security-critical workloads.

Overlay: Multi-host networking for Docker Swarm. Spans multiple Docker daemons. Used when you need containers on different machines to communicate.

Key point: Always use user-defined bridge networks (not the default bridge) — they provide automatic DNS resolution, better isolation, and can be managed independently.

6
How does Docker use Linux kernel features (namespaces, cgroups) for isolation?
Hard ›

Docker is not magic — it's a user-friendly wrapper around Linux kernel primitives:

Namespaces (isolation):

  • pid — container sees only its own processes (PID 1 = container process)
  • net — separate network stack, own IP, routing tables
  • mnt — separate mount namespace, container sees its own filesystem
  • uts — own hostname
  • ipc — separate inter-process communication
  • user — UID/GID mapping (optional)

cgroups (resource limits): Control groups limit and account for CPU, memory, disk I/O, network I/O. When you set --memory=512m, the kernel's OOM killer terminates the process if it exceeds that limit (Exit 137).

Union Filesystem (overlay2): Layers are stacked using overlay filesystem. Each layer is read-only. Container gets a writable layer on top. Changes in the container don't affect the image. Layers are shared between containers using the same image.

Why this matters: Containers share the HOST kernel. If the kernel has a vulnerability, all containers on that host are potentially affected. This is why VMs offer stronger security isolation.

7
What are Docker volumes vs bind mounts and when do you use each?
Medium ›

Named Volumes (-v mydata:/app/data): Docker manages the storage location (/var/lib/docker/volumes/). Portable, work on any platform, survive container deletion, better performance on Docker Desktop (Mac/Windows). Use for: databases, persistent app data in production.

Bind Mounts (-v $(pwd)/src:/app/src): Mount any host path into container. Changes on host immediately visible in container. Use for: local development (live code reload), sharing config files.

tmpfs: Mount in memory. Not persisted. Use for: sensitive temporary data, high-speed ephemeral storage.

Interview tip: "In production Docker Compose, I use named volumes for databases (postgres, redis) because they survive container recreations. In local development, I use bind mounts so code changes reflect instantly without rebuilding."

8
What are Docker security best practices for production images?
Hard ›

1. Never run as root: Add USER instruction. Running as root means a container escape gives full host access.

2. Use minimal base images: alpine, distroless, scratch. Fewer packages = smaller attack surface. Google's distroless images have no shell or package manager.

3. Pin image tags: Never use :latest. Pin to specific digest: FROM node:18.19.0-alpine3.19@sha256:abc123...

4. Scan images: Use AWS ECR image scanning (enabled by default), Trivy, or Snyk in CI pipeline.

5. Multi-stage builds: Remove build tools from production image.

6. .dockerignore: Exclude .git, .env, node_modules, secrets from build context.

7. Read-only filesystem: docker run --read-only (if app supports it)

8. Drop capabilities: docker run --cap-drop ALL --cap-add NET_BIND_SERVICE

9. No secrets in Dockerfile: Never COPY .env or hardcode passwords. Use ARG for build-time, Secrets for runtime.

☸️ Kubernetes Questions
9
Explain the Kubernetes architecture — control plane vs worker nodes.
Easy ›

Control Plane (runs the cluster brain):

  • kube-apiserver — REST API front door. All kubectl commands hit this. Validates and persists to etcd.
  • etcd — Distributed key-value store. Source of truth for all cluster state. Run as 3 or 5 nodes for HA.
  • kube-scheduler — Watches for unscheduled pods and assigns them to nodes based on resource availability, affinity rules, taints.
  • kube-controller-manager — Runs reconciliation loops: ReplicaSet controller, Node controller, Job controller.
  • cloud-controller-manager — AWS-specific: provisions ELBs, EBS volumes when K8s resources are created.

Worker Nodes (run the workloads):

  • kubelet — Agent on each node. Receives pod specs from API server, manages container lifecycle via container runtime.
  • kube-proxy — Maintains iptables/IPVS rules for Service networking.
  • Container runtime — containerd (or CRI-O). Actually runs containers.

On AWS EKS: AWS manages the control plane (you don't see those EC2s). You manage worker nodes (EC2 instances in a managed node group).

10
A pod is in CrashLoopBackOff. Walk me through your debugging process.
Medium ›

CrashLoopBackOff means: the container starts, crashes, K8s restarts it, it crashes again — repeatedly with exponential backoff (10s → 20s → 40s → 80s → 5min max).

Step-by-step debug process:

bash
# Step 1: Check pod status and restart count
kubectl get pods -n your-namespace
# Look for: RESTARTS column (high = CrashLoop)

# Step 2: Read logs from the CRASHED container
kubectl logs <pod-name> --previous -n your-namespace
# --previous is critical — shows logs from BEFORE the crash

# Step 3: Describe pod — check Events section
kubectl describe pod <pod-name> -n your-namespace
# Look for: OOMKilled, image pull errors, liveness probe failures

# Step 4: Check exit code
kubectl get pod <pod-name> -o json | jq '.status.containerStatuses[].lastState'
# Exit 1 = app error, Exit 137 = OOMKilled, Exit 139 = segfault

Common causes:

  • Application crash (unhandled exception, missing env var)
  • OOMKilled — memory limit too low (Exit 137)
  • Liveness probe too aggressive (killing healthy container)
  • Missing dependency (can't connect to DB)
  • Wrong CMD/ENTRYPOINT
11
What is the difference between a Deployment, StatefulSet, and DaemonSet?
Medium ›

Deployment: For stateless applications. Pods are interchangeable — any pod can handle any request. Rolling updates work by replacing pods. Use for: web servers, APIs, microservices.

StatefulSet: For stateful applications that need: (1) stable, predictable pod names (pod-0, pod-1, pod-2), (2) stable persistent storage (each pod gets its own PVC), (3) ordered startup/shutdown. Use for: databases (PostgreSQL, MySQL), Kafka, Zookeeper, Elasticsearch.

DaemonSet: Ensures ONE pod runs on EVERY node (or a subset). As nodes are added, a pod is scheduled automatically. As nodes are removed, pods are garbage collected. Use for: log collectors (Fluentd, Filebeat), monitoring agents (Node Exporter, Datadog), network plugins (Calico, AWS VPC CNI).

Job / CronJob: Job runs a task to completion (batch processing). CronJob runs on a schedule (nightly reports, DB backups).

12
Explain Kubernetes Services — when would you use ClusterIP vs LoadBalancer vs Ingress?
Hard ›

ClusterIP: Internal only. Used for service-to-service communication inside the cluster. Example: your web app talks to your postgres service via ClusterIP. No external access. DNS: postgres.default.svc.cluster.local

NodePort: Exposes service on a port (30000-32767) on every node. External access via NodeIP:NodePort. Avoid in production — requires knowing node IPs, exposes non-standard ports. Use only for testing.

LoadBalancer: Creates an AWS NLB/ELB. Good for non-HTTP workloads (TCP/UDP). Each service = one load balancer = ~$20/month. Costly if you have many services.

Ingress: HTTP/HTTPS routing rules. ONE load balancer (ALB) routes to many services based on host/path. Cost-effective for multiple HTTP services. Supports TLS termination, path-based routing, host-based routing. Requires an Ingress Controller (AWS Load Balancer Controller, nginx-ingress).

Decision tree: Internal service? → ClusterIP. External HTTP/HTTPS? → Ingress. External TCP/UDP? → LoadBalancer.

13
How does Kubernetes handle zero-downtime rolling updates? What can go wrong?
Hard ›

Rolling update process:

  1. Deployment creates a new ReplicaSet with the new image
  2. Gradually scales up new RS while scaling down old RS
  3. maxSurge controls max extra pods during update (default: 25%)
  4. maxUnavailable controls max pods that can be down (default: 25%)
  5. New pods must pass readinessProbe before old pods are removed

What can go wrong:

  • Missing readinessProbe — new pods become "ready" before app is actually ready → requests fail
  • readinessProbe too fast — app needs 20s to start but probe fires after 5s → pod killed too early
  • No preStop hook — in-flight requests dropped when pod receives SIGTERM → add preStop: sleep 5 to drain connections
  • Database migrations — new app version expects new schema but old version running during rollout → backward-compatible migrations are required
  • Resources too tight — cluster has no capacity for surge pods → add buffer capacity or set maxSurge:0
14
What is RBAC in Kubernetes and how does it work?
Medium ›

RBAC (Role-Based Access Control) controls WHO can do WHAT to WHICH resources in K8s.

Four objects:

  • Role — defines permissions within a namespace (get/list/create pods)
  • ClusterRole — defines permissions cluster-wide (or reusable across namespaces)
  • RoleBinding — grants a Role to a user/group/ServiceAccount in a namespace
  • ClusterRoleBinding — grants a ClusterRole cluster-wide
yaml
# Role: read-only access to pods in "staging" namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: staging
rules:
- apiGroups: [""]
  resources: ["pods", "pods/logs"]
  verbs: ["get", "list", "watch"]
---
# Bind to a developer
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: dev-pod-reader
  namespace: staging
subjects:
- kind: User
  name: alice@company.com
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

Interview tip: "ServiceAccounts are K8s identities for pods. Assign minimal RBAC permissions to each ServiceAccount (principle of least privilege). On EKS, use IAM Roles for Service Accounts (IRSA) to give pods AWS API access without storing credentials."

How does Kubernetes scheduling work? What is node affinity, taints, and tolerations?
Hard ›

Scheduling process: The kube-scheduler finds a node for each unscheduled pod by filtering (removes nodes that don't meet requirements) then scoring (ranks remaining nodes, picks highest).

Node Affinity: Attract pods TO nodes with certain labels. requiredDuringScheduling = hard rule (pod won't schedule if no match). preferredDuringScheduling = soft preference.

yaml
# Run only on nodes with GPU
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["g4dn.xlarge", "p3.2xlarge"]  # GPU instances

Taints: Mark nodes to REPEL pods. Nodes with a taint don't accept pods unless the pod has a matching toleration. Example: taint GPU nodes so only GPU workloads land there.

Tolerations: Allow pods to be scheduled on tainted nodes. "I tolerate this node's restrictions."

Pod Anti-Affinity: Spread pods AWAY from each other. Critical for HA — prevents all replicas landing on the same node.

16
What is a Kubernetes Namespace and why use them?
Medium ›

Namespaces are virtual clusters within a physical cluster. They provide logical isolation for different teams, environments, or applications.

Use cases:

  • Environment isolation: production, staging, development namespaces
  • Team isolation: frontend-team, backend-team, data-team namespaces
  • Resource quotas per namespace (prevent one team starving others)
  • RBAC scoped to namespace (developers can only access their namespace)

Important: Namespaces do NOT provide network isolation by default. A pod in namespace A can talk to a pod in namespace B using the full DNS name: service.namespace.svc.cluster.local. For network isolation, use NetworkPolicies.

Cluster-scoped resources (Nodes, PersistentVolumes, StorageClasses, ClusterRoles) don't belong to any namespace.

17
Design a highly available, production-grade K8s deployment. What would you include?
Hard ›

This is a system design question. Walk through each layer:

Cluster level: Multi-AZ node groups (3 AZs minimum). Cluster Autoscaler or Karpenter for node scaling. Separate node groups for different workload types (general, GPU, spot).

Application level: Minimum 3 replicas (one per AZ). PodAntiAffinity to spread across nodes. HPA for automatic scaling. PodDisruptionBudget (minAvailable: 2) to prevent K8s from taking down too many pods during maintenance.

Networking: ALB Ingress with health checks. Services use ClusterIP internally. NetworkPolicies for micro-segmentation.

Configuration: ConfigMaps for non-sensitive config. Secrets from AWS Secrets Manager via External Secrets Operator. Never hardcode credentials.

Storage: PVCs with gp3 StorageClass. For databases, prefer AWS RDS over in-cluster (better HA, backups, patches).

Observability: Prometheus + Grafana for metrics. Fluent Bit → CloudWatch for logs. AWS X-Ray or Jaeger for tracing. CloudWatch alarms + PagerDuty for on-call.

Security: IRSA for pod AWS access. RBAC with least privilege. Pod Security Standards (restricted). Network Policies. Image scanning in ECR.

⚙️ CI/CD Questions
18
What is CI/CD and why does it matter?
Easy ›

CI (Continuous Integration): Developers merge code frequently (multiple times a day). Every push triggers automated build and tests. Fast feedback — broken code is caught in minutes, not weeks.

CD (Continuous Delivery): Every successful CI build produces a deployable artifact. Deployment to production requires one click (or approval). Always in a releasable state.

CD (Continuous Deployment): Every passing build is automatically deployed to production. Zero human intervention. Requires excellent test coverage and monitoring.

Why it matters (DORA metrics): Elite teams deploy multiple times per day with less than 1 hour lead time and less than 5% change failure rate. Traditional teams deploy monthly with 30-90% failure rate. The business impact: faster feature delivery, fewer outages, better developer experience.

19
What is GitOps and how is it different from traditional CI/CD?
Medium ›

Traditional CI/CD: CI pipeline runs kubectl apply (push model). Pipeline needs cluster credentials. Manual kubectl changes can drift from what's in Git. No automatic reconciliation.

GitOps: Git is the single source of truth (pull model). A GitOps agent (ArgoCD, Flux) runs INSIDE the cluster. It continuously compares cluster state to Git state. Any drift is automatically corrected.

Benefits of GitOps:

  • Full audit trail — every cluster change is a git commit with author and message
  • Easy rollback — git revert to any previous state
  • No cluster credentials in CI — pipeline only updates Git
  • Self-healing — manual kubectl changes are reverted automatically
  • Multi-cluster management — same repo → multiple clusters

ArgoCD vs Flux: Both are CNCF GitOps tools. ArgoCD has a better UI and is more popular. Flux is more lightweight and Kubernetes-native. Both are production-grade.

20
How do you handle database migrations in a CI/CD pipeline with zero downtime?
Hard ›

This is one of the hardest problems in CD. During a rolling update, old and new app versions run simultaneously — both must work with the same database schema.

The rule: migrations must be backward-compatible.

Safe migration patterns (expand-contract):

  • Adding a column: Add nullable column → deploy new code that writes to it → backfill → add NOT NULL constraint. Never add a NOT NULL column with no default in one shot.
  • Renaming a column: Add new column → deploy code that writes both old+new → copy data → deploy code that reads new only → drop old column. Never rename directly.
  • Removing a column: Deploy code that doesn't read the column → remove column. Never remove a column that code still reads.

Implementation: Run migrations as a Kubernetes Job or initContainer before the main app starts. Use tools like Flyway or Liquibase. Lock migrations so only one pod runs them (advisory locks).

yaml
initContainers:
  - name: run-migrations
    image: myapp:v2.0.0
    command: ["npm", "run", "migrate"]
    envFrom:
      - secretRef:
          name: db-secret
21
How do you securely manage secrets in a CI/CD pipeline?
Medium ›

The anti-patterns to avoid:

  • Hardcoded credentials in code or Dockerfile
  • Secrets in environment variables that get logged
  • Long-lived AWS access keys stored in GitHub secrets
  • Secrets committed to Git (even accidentally, they're in history)

Production approach for GitHub Actions + AWS:

  • Use OIDC — GitHub Actions assumes an IAM role via federated identity. No static credentials stored anywhere.
  • Store app secrets in AWS Secrets Manager, not GitHub secrets
  • Use External Secrets Operator to sync Secrets Manager → K8s Secrets at deployment time
  • Rotate secrets regularly. Never share secrets between environments.

For local dev: Use .env files (never committed), or tools like direnv, 1Password CLI, or AWS SSM Parameter Store.

Git leak prevention: git-secrets, pre-commit hooks with detect-secrets, GitHub secret scanning.

☁️ AWS / EKS Questions
22
What is AWS EKS and what does AWS manage vs what do you manage?
Medium ›

AWS EKS is a managed Kubernetes service. AWS runs and maintains the control plane (kube-apiserver, etcd, kube-scheduler, controller-manager) across multiple AZs.

AWS manages: Control plane HA and upgrades, etcd backups, K8s API endpoint, certificate rotation, security patches for control plane.

You manage: Worker nodes (EC2 instances — unless using Fargate), node OS updates (managed node groups help), add-ons (CoreDNS, kube-proxy, VPC CNI), cluster add-ons (ALB controller, cluster autoscaler), your workloads.

EKS node options:

  • Managed node groups — AWS manages node lifecycle, OS updates. Most common.
  • Self-managed nodes — Full control, more operational overhead.
  • Fargate — Serverless pods. No node management. AWS provisions capacity per pod. Good for bursty, batch workloads. No DaemonSets, no privileged containers.
23
What is IRSA (IAM Roles for Service Accounts) and why is it important?
Hard ›

IRSA allows Kubernetes pods to assume AWS IAM roles without storing credentials. It uses OIDC federation between EKS and AWS IAM.

The problem it solves: Your pod needs to read from S3 or write to DynamoDB. Bad options: hardcode AWS keys in env vars (insecure), give EC2 node instance role S3 access (all pods on that node get access — over-permissioned).

How IRSA works:

  1. Each EKS cluster has an OIDC endpoint
  2. Create an IAM role with a trust policy that allows the specific K8s ServiceAccount to assume it
  3. Annotate the ServiceAccount with the IAM role ARN
  4. K8s injects a projected service account token into the pod
  5. AWS SDK in the pod exchanges the token for temporary AWS credentials
yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: myapp-sa
  namespace: production
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/myapp-s3-role

Benefits: Least privilege per pod, credentials automatically rotated, no stored secrets, full CloudTrail audit trail of pod API calls.

24
How do you monitor a Kubernetes cluster on AWS? What tools do you use?
Medium ›

Metrics: Install kube-state-metrics + Prometheus Operator. Grafana for dashboards. OR use Amazon Managed Prometheus + Amazon Managed Grafana for less operational overhead.

Logs: Fluent Bit DaemonSet collects container logs → CloudWatch Logs. Log Insights for querying. Or OpenSearch for more advanced search.

Tracing: AWS X-Ray or Jaeger for distributed tracing across microservices.

Alerting: CloudWatch Alarms → SNS → PagerDuty/OpsGenie for on-call. Alert on: pod crash rate, memory/CPU above 80%, HPA at max replicas, node NotReady, failed deployments.

Key metrics to monitor:

  • Pod restart count (CrashLoopBackOff detection)
  • CPU/memory requests vs limits vs actual usage
  • HPA current/min/max replicas
  • Node resource utilization
  • API server latency and error rate
  • PVC capacity

Container Insights: AWS-native CloudWatch feature for EKS — auto-collects metrics without manual Prometheus setup. Good starting point.

25
How would you set up autoscaling for an EKS cluster (pods AND nodes)?
Hard ›

Two levels of autoscaling:

1. Pod autoscaling (HPA): Scales pods based on CPU/memory/custom metrics. Configure minReplicas, maxReplicas, targetCPUUtilizationPercentage. Requires metrics-server installed.

2. Node autoscaling: When HPA wants to add pods but no node has capacity, nodes must be added. Two options:

Cluster Autoscaler: AWS-specific. Monitors for pending pods. Scales up the node group. Scales down underutilized nodes after 10min (configurable). Simple, battle-tested.

Karpenter: Newer, AWS-built. Provisions nodes directly (not through ASG). Faster scale-up (30s vs 3min for CA). Bin-packs pods efficiently. Supports diverse instance types (picks cheapest). Consolidates underutilized nodes automatically. Generally preferred for new EKS clusters.

Complete setup:

  1. Install metrics-server (required for HPA)
  2. Create HPA for each deployment
  3. Install Karpenter (or Cluster Autoscaler)
  4. Create Karpenter NodePool with instance type constraints
  5. Test: kubectl run load-generator — watch HPA scale pods, then Karpenter provision nodes
