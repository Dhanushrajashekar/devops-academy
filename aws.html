AWS Overview & Core Services

The 20% of AWS that powers 80% of DevOps workloads — mapped to real job requirements.

Foundation~25 min

☁️ Why AWS Dominates DevOps

AWS holds ~32% of the cloud market. Most DevOps job descriptions list EC2, S3, IAM, and EKS. You don't need to know all 200+ AWS services — master the core 12 and you're interview-ready.

When an interviewer asks "how would you make this highly available?" — the answer always involves deploying across multiple AZs. Spread your EC2 instances, RDS replicas, and EKS nodes across at least 2 AZs so one data center failure doesn't take you down.

IAM — Identity & Access Management

The security foundation of everything in AWS. Get this wrong and you get breached.

Critical~30 min

The Four IAM Entities

EntityWhat It IsWhen to Use
UserA person or service with long-term credentials (username + password, access keys)Human admins. Avoid for applications — use roles instead.
GroupCollection of users. Policies attached to group apply to all members.Organize permissions: Developers group, Ops group, ReadOnly group.
RoleAssumed by entities (EC2, Lambda, EKS pods, CI/CD). Provides temporary credentials.EC2 instance role, EKS node role, GitHub Actions OIDC role, cross-account access.
PolicyJSON document defining allowed/denied actions on resources.Attached to Users, Groups, or Roles to grant permissions.

IAM Policy Structure

{ "Version": "2012-10-17", "Statement": [ { "Sid": "AllowS3ReadBucket", "Effect": "Allow", // Allow or Deny "Action": [ "s3:GetObject", "s3:ListBucket" ], "Resource": [ "arn:aws:s3:::my-app-bucket", "arn:aws:s3:::my-app-bucket/*" ] } ] }
⚠️
Least Privilege — The Golden Rule
Grant only the permissions needed, nothing more. An EC2 instance that only reads from S3 should have s3:GetObject only — not s3:* or *:*. Most AWS breaches start with overly-permissive IAM policies.

IAM Roles vs Access Keys

MethodCredentialsRecommended?Use Case
IAM RoleTemporary STS credentials (15 min – 12 hr)✅ YesEC2 instance role, EKS node/pod role, Lambda execution role
OIDC FederationTemp credentials via identity provider✅ YesGitHub Actions assuming AWS role without long-term keys
Access KeysLong-term key ID + secret (never expire)⚠️ AvoidLocal CLI dev only. Rotate frequently. Never commit to code.

IRSA — IAM Roles for Service Accounts (EKS)

IRSA lets EKS pods assume IAM roles without node-level credentials. The pod gets its own role and only has the permissions it needs.

# 1. Create IAM OIDC provider for the EKS cluster eksctl utils associate-iam-oidc-provider \ --cluster my-cluster \ --approve # 2. Create IAM role for the service account eksctl create iamserviceaccount \ --cluster my-cluster \ --namespace my-app \ --name my-app-sa \ --attach-policy-arn arn:aws:iam::123456789:policy/MyAppPolicy \ --approve # 3. Annotate the Kubernetes service account kubectl annotate serviceaccount my-app-sa -n my-app \ eks.amazonaws.com/role-arn=arn:aws:iam::123456789:role/my-app-role

GitHub Actions → AWS (Keyless Auth)

# .github/workflows/deploy.yml permissions: id-token: write # Required for OIDC contents: read jobs: deploy: steps: - name: Configure AWS credentials uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: arn:aws:iam::123456789:role/github-actions-role aws-region: us-east-1 # No access keys needed — OIDC generates temp credentials

VPC — Virtual Private Cloud

Your isolated network in AWS. Understanding VPCs is required for any senior DevOps role.

Critical~35 min

VPC Architecture

Public vs Private Subnets

PropertyPublic SubnetPrivate Subnet
Route to internetDirect via Internet Gateway (IGW)Outbound only via NAT Gateway
Has public IP?Yes (Elastic IP or auto-assign)No public IPs
What lives hereALB, NAT Gateway, Bastion HostEC2 app servers, EKS nodes, RDS databases
Internet can reach?Yes (with security group rules)No — only reachable via internal routing
🎯
Interview: Why put EKS nodes in private subnets?
Answer: Security. Your application pods should never be directly reachable from the internet. Traffic flows: Internet → ALB (public subnet) → EKS nodes (private subnet). NAT Gateway allows nodes to pull container images (egress-only). This is the standard production architecture.

Adding: "We also set security groups so only the ALB security group can reach the node group on port 80/443."

Purchasing Options — Cost Optimization

OptionDiscount vs On-DemandCommitmentBest For
On-DemandNo discount (baseline)NoneVariable workloads, spiky traffic
Reserved InstancesUp to 72% off1-3 yearsStable baseline capacity (e.g., always-on EKS nodes)

💡
Production EKS Node Strategy
Use a mixed node group: On-Demand instances for your baseline (1-2 per AZ) to guarantee capacity, and Spot instances for burst capacity (Karpenter auto-selects cheapest available spot type). This cuts EKS compute cost by 50-70% while maintaining reliability.

Key EC2 Concepts for Interviews

AMI
  • Amazon Machine Image — the OS template
  • Amazon Linux 2023 for EKS nodes
  • Build custom AMIs with Packer
  • Immutable infra: bake don't configure
Auto Scaling Group
  • Maintain min/max/desired instance count
  • Scale based on CPU, custom metrics
  • Replace unhealthy instances automatically
  • EKS managed node groups use ASGs
EBS Volumes
  • Block storage attached to EC2
  • gp3 = default, best price/performance
  • io2 = provisioned IOPS (databases)
  • Snapshots for backup to S3

S3 in DevOps Workflows

# Terraform remote state in S3 terraform { backend "s3" { bucket = "my-company-tfstate" key = "production/eks/terraform.tfstate" region = "us-east-1" dynamodb_table = "terraform-state-lock" # Prevents concurrent apply encrypt = true } } # Upload build artifacts from CI aws s3 cp dist/ s3://my-app-artifacts/${GITHUB_SHA}/ --recursive # Sync static website aws s3 sync build/ s3://my-app-static-site/ --delete # Cross-account access with bucket policy aws s3api put-bucket-policy --bucket my-bucket \ --policy file://cross-account-policy.json

S3 Security Checklist

🔒
S3 Security Must-Haves
✅ Block Public Access — enable on all buckets unless serving a public website
✅ Bucket versioning — for Terraform state and critical data
✅ Server-side encryption — SSE-S3 (default) or SSE-KMS for compliance
✅ Access logging — log all bucket access to another bucket
✅ S3 Object Lock — WORM protection for compliance archives
❌ Never make S3 buckets public unless explicitly required (static website)

Create EKS Cluster with eksctl

# cluster.yaml — production-ready config apiVersion: eksctl.io/v1alpha5 kind: ClusterConfig metadata: name: production region: us-east-1 version: "1.29" vpc: subnets: private: us-east-1a: { id: subnet-0abc123 } us-east-1b: { id: subnet-0def456 } public: us-east-1a: { id: subnet-0ghi789 } us-east-1b: { id: subnet-0jkl012 } managedNodeGroups: - name: workers instanceType: m6i.xlarge minSize: 2 maxSize: 10 desiredCapacity: 3 privateNetworking: true # Nodes in private subnets iam: withAddonPolicies: albIngress: true cloudWatch: true ebs: true # Create cluster (takes ~15 minutes) eksctl create cluster -f cluster.yaml

Essential EKS Add-ons

AWS Load Balancer Controller
  • Provisions ALB for Ingress resources
  • Provisions NLB for LoadBalancer Services
  • Replaces deprecated in-tree ELB controller
EBS CSI Driver
  • Dynamic PersistentVolume provisioning
  • Mount EBS volumes into pods
  • Required for stateful workloads on EKS
Karpenter
  • Replaces Cluster Autoscaler
  • Provisions nodes in <60 seconds
  • Selects cheapest/best-fit instance type
CoreDNS + kube-proxy
  • CoreDNS: service discovery in cluster
  • kube-proxy: network rules on nodes
  • Both managed as EKS addons

ECR — Elastic Container Registry

Private Docker registry for your container images, tightly integrated with EKS.

Core~15 min

ECR Workflow

# 1. Authenticate Docker to ECR aws ecr get-login-password --region us-east-1 | \ docker login --username AWS --password-stdin \ 123456789.dkr.ecr.us-east-1.amazonaws.com # 2. Build and tag image docker build -t my-app:${GITHUB_SHA} . docker tag my-app:${GITHUB_SHA} \ 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:${GITHUB_SHA} # 3. Push to ECR docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:${GITHUB_SHA} # 4. EKS pulls automatically (node IAM role grants ECR read access) # In your Deployment spec: image: 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:${GITHUB_SHA}

ECR Features

Image Scanning
  • Scan on push for CVEs (Basic or Enhanced)
  • Enhanced scanning uses Amazon Inspector
  • Block pipeline on CRITICAL findings
Lifecycle Policies
  • Auto-delete old untagged images
  • Keep last N tagged images
  • Prevents registry from growing unbounded
Cross-Account Access
  • ECR resource-based policies
  • Share images across AWS accounts
  • ECR Public for open-source images
💡
Why use ECR instead of Docker Hub?
Interview answer:
1. IAM authentication — no stored Docker Hub credentials in CI
2. No pull rate limits — Docker Hub throttles anonymous pulls
3. Same VPC — EKS node pulls from ECR without hitting the internet
4. Integrated scanning — Amazon Inspector scans images for vulnerabilities
5. Cost — No egress fees within the same region

Define AWS infrastructure in code — versioned, repeatable, auditable.

Terraform: EKS Cluster Module

# main.tf — Production EKS with Terraform module "eks" { source = "terraform-aws-modules/eks/aws" version = "~> 20.0" cluster_name = "production" cluster_version = "1.29" vpc_id = module.vpc.vpc_id subnet_ids = module.vpc.private_subnets # Managed node group eks_managed_node_groups = { workers = { instance_types = ["m6i.xlarge"] min_size = 2 max_size = 10 desired_size = 3 } } # Enable IRSA enable_irsa = true } # Remote state backend terraform { backend "s3" { bucket = "my-tfstate-bucket" key = "production/eks/terraform.tfstate" region = "us-east-1" dynamodb_table = "terraform-state-lock" encrypt = true } } # Deploy workflow terraform init # Download providers terraform plan # Preview changes (always run first!) terraform apply # Apply (prompts for confirmation) terraform destroy # Tear down (dangerous — confirm carefully)
⚠️
Never run terraform apply without terraform plan first
Always review the plan output. Watch for forces replacement — this means the resource will be destroyed and recreated. For databases and EKS clusters, this can cause downtime. Use -target to apply specific resources when needed.

Shipping EKS Logs to CloudWatch

# Deploy Fluent Bit as DaemonSet to ship pod logs helm repo add fluent https://fluent.github.io/helm-charts helm upgrade --install fluent-bit fluent/fluent-bit \ --namespace logging \ --create-namespace \ --set config.outputs="[OUTPUT] Name cloudwatch_logs Match * region us-east-1 log_group_name /aws/eks/production/apps log_stream_prefix pod- auto_create_group On"
# Create alarm: alert when pod error rate > 5% aws cloudwatch put-metric-alarm \ --alarm-name "high-error-rate" \ --metric-name "HTTPCode_Target_5XX_Count" \ --namespace "AWS/ApplicationELB" \ --statistic Sum \ --period 60 \ --threshold 50 \ --comparison-operator GreaterThanThreshold \ --evaluation-periods 2 \ --alarm-actions arn:aws:sns:us-east-1:123456789:pagerduty-alert

🔬 Lab: Deploy EKS in a Production VPC

Full hands-on: VPC with public/private subnets → EKS cluster → ALB → app running.

Lab~90 min
💰
AWS Costs — Estimated $3–$5 for this lab
An EKS control plane costs $0.10/hour (~$2.40/day). 2x t3.medium nodes add ~$0.08/hour. Run the cleanup commands at the end to avoid ongoing charges. A free alternative is to use AWS Free Tier with a local minikube for practice.

Step 1: Prerequisites

# Install required tools # AWS CLI v2 curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip" unzip awscliv2.zip && sudo ./aws/install aws configure # Enter: Access Key, Secret Key, region (us-east-1), output (json) # eksctl curl --silent --location "https://github.com/eksctl-io/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp sudo mv /tmp/eksctl /usr/local/bin # kubectl curl -LO "https://dl.k8s.io/release/v1.29.0/bin/linux/amd64/kubectl" sudo install kubectl /usr/local/bin/kubectl # helm curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

Step 2: Create VPC + EKS Cluster

# cluster.yaml cat > cluster.yaml <<'EOF' apiVersion: eksctl.io/v1alpha5 kind: ClusterConfig metadata: name: academy-demo region: us-east-1 version: "1.29" managedNodeGroups: - name: workers instanceType: t3.medium minSize: 2 maxSize: 4 desiredCapacity: 2 iam: withAddonPolicies: albIngress: true cloudWatch: true ebs: true EOF # Create cluster (~15 minutes) eksctl create cluster -f cluster.yaml # Verify kubectl get nodes

Step 3: Deploy Sample App + ALB Ingress

# Install AWS Load Balancer Controller helm repo add eks https://aws.github.io/eks-charts helm upgrade --install aws-load-balancer-controller eks/aws-load-balancer-controller \ -n kube-system \ --set clusterName=academy-demo \ --set serviceAccount.create=false \ --set serviceAccount.name=aws-load-balancer-controller # Deploy app with Ingress kubectl apply -f - <<'EOF' apiVersion: apps/v1 kind: Deployment metadata: name: demo-app spec: replicas: 2 selector: matchLabels: app: demo template: metadata: labels: app: demo spec: containers: - name: app image: nginx:alpine ports: - containerPort: 80 --- apiVersion: v1 kind: Service metadata: name: demo-svc spec: selector: app: demo ports: - port: 80 --- apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: demo-ingress annotations: kubernetes.io/ingress.class: alb alb.ingress.kubernetes.io/scheme: internet-facing alb.ingress.kubernetes.io/target-type: ip spec: rules: - http: paths: - path: / pathType: Prefix backend: service: name: demo-svc port: number: 80 EOF # Get ALB URL (takes 2-3 minutes to provision) kubectl get ingress demo-ingress -w

Step 4: Cleanup (Important!)

# Delete all resources to stop billing kubectl delete ingress demo-ingress # Delete ALB first (otherwise eksctl delete hangs) eksctl delete cluster --name academy-demo --region us-east-1
