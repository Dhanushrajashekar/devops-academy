What is Docker?

Understand the problem Docker solves and why the industry adopted it universally.

Beginner ~20 min

The Problem Before Docker

Before Docker, deploying software was painful. A developer would build an app on their Mac, it would work perfectly — then it would fail in production on Linux because of different Python versions, missing libraries, or different OS configurations. "It works on my machine" was a constant refrain.

Container vs Virtual Machine

Containers and VMs both isolate applications, but they work very differently. This is a fundamental concept you will be asked about in interviews.

FeatureContainer (Docker)Virtual Machine
Startup timeMillisecondsMinutes
SizeMegabytesGigabytes
OS isolationShares host OS kernelFull OS per VM
Resource overheadVery lowHigh (full OS)
Security isolationProcess-level namespacesHardware-level hypervisor
PortabilityRun anywhere Docker existsHypervisor-dependent
Use caseMicroservices, CI/CD, scalingFull OS needed, legacy apps
💡
Interview Answer: Containers vs VMs
Say: "Containers share the host OS kernel and use Linux namespaces and cgroups for isolation. VMs run a full guest OS on a hypervisor. Containers are faster and smaller, but VMs offer stronger isolation. In practice, AWS uses both — EC2 nodes run VMs, and inside those nodes, Kubernetes runs containers."

How Docker Works — The 3 Core Concepts

1. Dockerfile

A text file with instructions to build an image. Think of it as a recipe.

2. Image

A read-only snapshot of your application + all dependencies. Built from a Dockerfile. Stored in a registry (Docker Hub, AWS ECR). Like a class in OOP.

3. Container

A running instance of an image. You can have many containers from one image. Like an object (instance) of a class.

Docker Architecture

How Docker Daemon, Client, Images, and Registry work together.

Core Concept Interview Must-Know

The Docker Client-Server Model

Docker uses a client-server architecture. When you type docker run nginx, here is exactly what happens step by step:

What Happens When You Run: docker run nginx

  1. Docker CLI sends the command to Docker Daemon via REST API over a Unix socket
  2. Docker Daemon checks if the nginx image exists locally
  3. If not found locally → pulls from Docker Hub (or your configured registry)
  4. Daemon calls containerd to create a container from the image
  5. containerd calls runc to set up Linux namespaces and cgroups
  6. Container starts! The nginx process runs in an isolated environment

Linux Primitives Docker Uses

Docker is not magic — it's Linux kernel features exposed in a developer-friendly way:

Linux FeatureWhat Docker Uses It ForExample
NamespacesProcess isolation (PID, NET, MNT, UTS, IPC)Container can't see host processes
cgroupsResource limits (CPU, memory)--memory=512m --cpus=1
Union FSLayered image filesystem (overlay2)Shared base layers, fast builds
chrootIsolated root filesystemContainer sees its own / directory
🎯
Interview Gold: "How does Docker isolate containers?"
Answer: "Docker uses Linux namespaces for isolation (each container gets its own PID, network, and mount namespaces) and cgroups to limit resource usage. The container shares the host kernel but cannot see other processes or the host filesystem. The overlay2 storage driver enables efficient copy-on-write layering of images."

Installation & First Run

Install Docker and run your first container on every major platform.

Hands-On

Install Docker on Ubuntu / Debian (EC2 or local)

bashUbuntu install
# 1. Remove old versions
sudo apt-get remove docker docker-engine docker.io containerd runc

# 2. Install prerequisites
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release

# 3. Add Docker's official GPG key
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# 4. Set up the stable repository
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# 5. Install Docker Engine
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# 6. Add your user to docker group (avoid sudo every time)
sudo usermod -aG docker $USER
newgrp docker

# 7. Verify installation
docker run hello-world

Images & Containers

Understanding the difference and working with Docker Hub images.

Core ConceptHands-On

Docker Images are Made of Layers

A Docker image is not a single file — it's a stack of read-only layers. Each instruction in a Dockerfile creates a layer. Layers are shared between images, which makes them fast and storage-efficient.

Pulling & Managing Images

⌨️
Type these commands:
docker pull nginx — download nginx image
docker pull node:18 — specific tag (always pin tags in production!)
docker images — list all local images
docker rmi nginx — remove an image
docker system prune — clean up unused resources

Image Naming Convention

text
# Format: [registry/][namespace/]repository[:tag][@digest]

nginx                              # Docker Hub official image, latest tag
nginx:1.25.3                       # Specific version (USE THIS in production!)
node:18-alpine                     # node v18, alpine variant (tiny)
mycompany/myapp:v2.1.0             # Custom image on Docker Hub
123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest   # AWS ECR
ghcr.io/myorg/myapp:sha-abc123     # GitHub Container Registry
⚠️
Never use :latest in production
Using nginx:latest means your image changes whenever Docker Hub updates nginx. This breaks reproducibility. Always pin to a specific version: nginx:1.25.3. This is a real mistake companies make and a common interview question.

Writing Production Dockerfiles

Every instruction, every best practice. Real examples for Node.js, Python, Go.

Industry Critical Hands-On

Every Dockerfile Instruction Explained

dockerfileComplete Dockerfile reference
# FROM — base image (ALWAYS first instruction)
FROM node:18-alpine

# ARG — build-time variables (not available at runtime)
ARG APP_VERSION=1.0.0

# ENV — environment variables (available at runtime too)
ENV NODE_ENV=production
ENV PORT=3000

# WORKDIR — sets working directory (creates it if missing)
WORKDIR /app

# COPY — copy files from host to image
# COPY src dest   (relative to WORKDIR)
COPY package*.json ./          # Copy FIRST for layer caching

# RUN — execute command during build (creates a new layer)
RUN npm ci --only=production   # npm ci is faster and more reliable than npm install

# COPY the rest of the app (separate layer — cache deps above)
COPY . .

# EXPOSE — document which port the container listens on
# Does NOT actually publish the port — use -p flag in docker run
EXPOSE 3000

# USER — run as non-root (security best practice!)
USER node

# HEALTHCHECK — Docker monitors container health
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

# CMD — default command to run (can be overridden)
CMD ["node", "server.js"]

# ENTRYPOINT — cannot be overridden (use for the main process)
# ENTRYPOINT ["node"] + CMD ["server.js"] = node server.js

Real Production Examples

Production Node.js Dockerfile

dockerfileDockerfile (Node.js)
FROM node:18-alpine AS base

# Install dumb-init for proper signal handling
RUN apk add --no-cache dumb-init

WORKDIR /app

# Copy package files FIRST (layer caching trick)
COPY package*.json ./

# Install only production deps
RUN npm ci --only=production && npm cache clean --force

# Copy source code
COPY . .

# Don't run as root!
USER node

EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

# dumb-init handles PID 1 and signal forwarding properly
ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "server.js"]

Production Go Dockerfile (Multi-Stage)

dockerfileDockerfile (Go)
# Stage 1: Build
FROM golang:1.21-alpine AS builder

WORKDIR /app

# Download deps first (layer cache)
COPY go.mod go.sum ./
RUN go mod download

COPY . .

# Build static binary
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-w -s" -o /myapp ./cmd/server

# Stage 2: Minimal production image (scratch = empty!)
FROM scratch

# Copy SSL certs for HTTPS
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

# Copy ONLY the binary
COPY --from=builder /myapp /myapp

EXPOSE 8080
ENTRYPOINT ["/myapp"]

# Result: ~5-10MB image vs ~1GB with full Go image!

Spring Boot Dockerfile

⚠️
Top 5 Dockerfile Mistakes (and Fixes)
1. No .dockerignore → adds node_modules (1GB!) to build context
2. Running as root → security vulnerability, use USER
3. COPY . . before npm install → breaks layer caching, slow rebuilds
4. Installing dev dependencies → bloats production image
5. Not pinning base image tags → non-reproducible builds

Multi-Stage Builds

Reduce image size by 90%+. The industry standard for production images.

Production Must-Know Interview Favorite

Why Multi-Stage?

Building a Node.js app requires compilers, test tools, and build utilities. Running it in production needs none of that. Multi-stage builds let you use a fat image to build, then copy only what's needed into a tiny final image.

ApproachImage SizeSecurityBuild Tools in Prod
Single stage (node:18)~1GBPoorYes (npm, compilers)
Single stage (node:18-alpine)~250MBOKYes
Multi-stage~80MBExcellentNo
Multi-stage (Go/scratch)~5-10MBBestNo

Complete Multi-Stage Example

dockerfileMulti-stage Node.js + React
# ===== STAGE 1: Build React frontend =====
FROM node:18-alpine AS frontend-builder

WORKDIR /app/frontend

COPY frontend/package*.json ./
RUN npm ci

COPY frontend/ .
RUN npm run build              # Creates /app/frontend/dist

# ===== STAGE 2: Build backend =====
FROM node:18-alpine AS backend-builder

WORKDIR /app

COPY backend/package*.json ./
RUN npm ci --only=production

COPY backend/ .

# ===== STAGE 3: Production image =====
FROM node:18-alpine

RUN apk add --no-cache dumb-init

WORKDIR /app

# Copy ONLY what's needed from previous stages
COPY --from=backend-builder /app/node_modules ./node_modules
COPY --from=backend-builder /app/src ./src
COPY --from=frontend-builder /app/frontend/dist ./public

USER node
EXPOSE 3000

ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "src/server.js"]

# Build: docker build -t myapp:latest .
# node_modules from Stage 1 & 2 are NOT in this image!
# Result: ~80MB instead of ~1.5GB
⌨️
Build and check image sizes:
docker build -t myapp:v1 .
docker images — compare sizes

How Bridge Networking Works

bash
# Create a custom bridge network
docker network create myapp-net

# Start containers on the same network
docker run -d --name postgres --network myapp-net \
  -e POSTGRES_PASSWORD=secret postgres:15

docker run -d --name redis --network myapp-net redis:7

docker run -d --name web --network myapp-net \
  -p 3000:3000 \
  -e DATABASE_URL=postgresql://postgres:secret@postgres:5432/mydb \
  -e REDIS_URL=redis://redis:6379 \
  myapp:latest

# ✓ 'web' can reach 'postgres' using just the container name 'postgres'
# ✓ Docker provides DNS resolution between containers on same network
# ✓ Only port 3000 is exposed to the host

Port Mapping Explained

bash
# -p HOST_PORT:CONTAINER_PORT
docker run -p 80:80 nginx        # host:80 → container:80
docker run -p 8080:80 nginx      # host:8080 → container:80
docker run -p 127.0.0.1:80:80 nginx  # only localhost, not external

# Multiple ports
docker run -p 80:80 -p 443:443 nginx

# Random host port (Docker assigns it)
docker run -p 80 nginx           # check with: docker ps

Volumes & Data Persistence

Container data is ephemeral by default. Learn how to persist it correctly.

Core Concept

Why Volumes?

When a container is deleted, all its data is gone. For databases, uploads, and logs, you need persistent storage.

Storage TypeCommandUse CaseManaged By
Volume-v mydata:/var/lib/mysqlDatabases, productionDocker
Bind Mount-v $(pwd)/src:/app/srcLocal developmentHost filesystem
tmpfs--tmpfs /tmpSensitive temp dataRAM

Complete Volume Examples

bash
# Create a named volume
docker volume create db-data

# Run PostgreSQL with persistent data volume
docker run -d \
  --name postgres \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=myapp \
  -v db-data:/var/lib/postgresql/data \     # named volume
  -p 5432:5432 \
  postgres:15

# Delete and recreate the container — data PERSISTS
docker rm -f postgres
docker run -d --name postgres -v db-data:/var/lib/postgresql/data postgres:15
# Your data is still there!

# Bind mount for development (live code reload)
docker run -d \
  --name myapp-dev \
  -v $(pwd):/app \                          # bind mount — code changes reflect instantly
  -v /app/node_modules \                    # anonymous volume for node_modules
  -p 3000:3000 \
  myapp:dev

# List all volumes
docker volume ls

# Inspect volume location on host
docker volume inspect db-data

# Backup a volume
docker run --rm \
  -v db-data:/data \
  -v $(pwd):/backup \
  alpine tar czf /backup/db-backup.tar.gz /data

Docker Compose

Define and run multi-container applications. The standard for local development.

Industry Standard Hands-On

Complete docker-compose.yml Reference

yamldocker-compose.yml — full production-like example
version: '3.9'

services:
  # ===== WEB APPLICATION =====
  web:
    build:
      context: .
      dockerfile: Dockerfile
      target: production   # multi-stage target
    image: myapp:latest
    container_name: myapp-web
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - PORT=3000
      - DATABASE_URL=postgresql://postgres:${DB_PASSWORD}@db:5432/myapp
      - REDIS_URL=redis://redis:6379
    env_file:
      - .env          # load from .env file
    depends_on:
      db:
        condition: service_healthy  # wait for healthy check
      redis:
        condition: service_healthy
    networks:
      - backend
      - frontend
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  # ===== POSTGRESQL DATABASE =====
  db:
    image: postgres:15-alpine
    container_name: myapp-db
    restart: unless-stopped
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: myapp
    volumes:
      - db-data:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # init script
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  # ===== REDIS CACHE =====
  redis:
    image: redis:7-alpine
    container_name: myapp-redis
    restart: unless-stopped
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis-data:/data
    networks:
      - backend
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3

  # ===== NGINX REVERSE PROXY =====
  nginx:
    image: nginx:1.25-alpine
    container_name: myapp-nginx
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./ssl:/etc/nginx/ssl:ro
    depends_on:
      - web
    networks:
      - frontend

volumes:
  db-data:        # persists PostgreSQL data
  redis-data:     # persists Redis data

networks:
  backend:        # internal — web, db, redis
    internal: true  # no external access!
  frontend:       # nginx → web

Push Images to AWS ECR

Create a private registry on AWS and push your Docker images. This is what production looks like.

AWS Real Cloud Hands-On Lab
☁️
Prerequisites for This Lab
AWS account, AWS CLI installed and configured (aws configure), Docker running. This is real cloud — you'll see actual ECR repositories created in your AWS console.

Step 1: Set Up AWS CLI

bash
# Install AWS CLI (if not installed)
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install

# Configure with your AWS credentials
aws configure
# Enter: Access Key ID, Secret Access Key, Region (us-east-1), Output format (json)

# Verify
aws sts get-caller-identity

Step 2: Create ECR Repository

bash
# Set your AWS account ID and region
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
AWS_REGION=us-east-1

# Create ECR repository
aws ecr create-repository \
  --repository-name myapp \
  --region $AWS_REGION \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=AES256

# Output shows the repositoryUri — save this!
# 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp

Step 3: Authenticate Docker to ECR

bash
# Login to ECR (tokens expire after 12 hours) aws ecr get-login-password --region $AWS_REGION | \ docker login --username AWS \ --password-stdin ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com # You should see: Login Succeeded

Step 4: Build, Tag & Push

bash
# Define your ECR URI
ECR_URI=${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/myapp

# Build the image
docker build -t myapp:latest .

# Tag it for ECR (two tags: latest + version)
docker tag myapp:latest ${ECR_URI}:latest
docker tag myapp:latest ${ECR_URI}:v1.0.0

# Push both tags
docker push ${ECR_URI}:latest
docker push ${ECR_URI}:v1.0.0

# Verify it's in ECR
aws ecr describe-images --repository-name myapp --region $AWS_REGION

Step 5: ECR Lifecycle Policy (Clean Old Images)

jsonlifecycle-policy.json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep last 10 images per tag prefix",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Remove untagged images after 1 day",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 1
      },
      "action": { "type": "expire" }
    }
  ]
}
bash
aws ecr put-lifecycle-policy \
  --repository-name myapp \
  --lifecycle-policy-text file://lifecycle-policy.json

Push to AWS ECR
›
bash
# Get your account ID
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
AWS_REGION=us-east-1
ECR_URI=${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

# Create ECR repository
aws ecr create-repository --repository-name myapp --region $AWS_REGION

# Login to ECR
aws ecr get-login-password --region $AWS_REGION | \
  docker login --username AWS --password-stdin ${ECR_URI}

# Tag for ECR
docker tag myapp:v1.0.0 ${ECR_URI}/myapp:v1.0.0
docker tag myapp:v1.0.0 ${ECR_URI}/myapp:latest

# Push!
docker push ${ECR_URI}/myapp:v1.0.0
docker push ${ECR_URI}/myapp:latest

# Verify in ECR
aws ecr list-images --repository-name myapp

bash
# Stop and remove container
docker stop myapp && docker rm myapp

# Remove local images (they're safe in ECR)
docker rmi myapp:v1.0.0

# Clean unused resources
docker system prune -f

# If you want to delete ECR repo (to avoid AWS costs)
aws ecr delete-repository \
  --repository-name myapp \
  --force \
  --region $AWS_REGION
